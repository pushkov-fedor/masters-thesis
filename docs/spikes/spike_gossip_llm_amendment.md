## 0. Контекст и причина amendment

В основном `spike_gossip.md` §8 (Recommendation) и §12 Q-J7 я предложил **gossip только в параметрическом симуляторе; LLM-симулятор gossip не получает**. Аргументация была три-составная:

1. По аналогии с accepted Q-G в `spike_llm_simulator.md` («capacity в промпт LLM-агента не передавать»).
2. Сохранение «независимости» двух симуляторов (PROJECT_DESIGN §7).
3. Cross-validation как «робастность ранжирования политик», не как «воспроизведение механизма».

Пользователь не согласен с этим выводом. Уточнённая позиция:

> «Gossip должен быть и в параметрическом симуляторе, и в LLM-симуляторе. При этом я не говорю автоматически делать полный OASIS / AgentSociety / RecAgent-style social world. Нужно исследовать варианты от простого агрегированного social signal в промпте до OASIS-inspired controlled social layer и более тяжёлых — последние, возможно, отклонить.»

Этот amendment пересматривает решение по LLM-gossip, не переписывая остальное в `spike_gossip.md`. Содержательно меняются: §8 Recommendation (LLM-сторона), §10 пункт 5 (откладываемое), §12 Q-J7 (полностью переформулирован). Параметрическая часть (V5 log_count, симплексная нормировка, EC-проверки L.1–L.6) **остаётся без изменений** — это важно: amendment не отменяет accepted-кандидата для параметрического симулятора, он только добавляет LLM-сторону.

## 1. Где встроить gossip в LLM-симуляторе

### 1.1. Текущая архитектура LLMAgent

Файл: `experiments/src/llm_agent.py`. Класс `LLMAgent` (строки 53–146):

- `agent_id: str`, `profile: str`, `history: list[dict]`.
- `decide(slot_id, talks, hall_loads_pct, recommendation, llm_call)` — точка вкладывания. Параметр `hall_loads_pct: dict[hall_id → float in 0..>1]` сейчас в сигнатуре, **но не используется в промпте** (docstring строки 75–77 явно фиксирует решение Q-G accepted в spike_llm_simulator).
- `USER_PROMPT_TEMPLATE` (строки 31–41) содержит блоки: `профиль`, `история сегодня`, `варианты в слоте`, опциональная `рекомендация системы`. **Социального сигнала нет.**
- Системный промпт `SYSTEM_PROMPT` (строки 22–28) перечисляет три фактора решения: профиль, история, рекомендация. Никакого «учитывай, что выбирают другие».

### 1.2. Где можно встроить gossip-сигнал

Минимально-инвазивно — расширить `USER_PROMPT_TEMPLATE` дополнительным блоком и/или передавать сводку `gossip_summary` в `decide()`:

```python
# Текущий USER_PROMPT_TEMPLATE: профиль + история + варианты + recommendation.
# Новое (для всех L1-L5): + опциональный блок «социальный сигнал».
GOSSIP_BLOCK_TEMPLATE = "\nЧто уже выбирают другие участники в этом слоте:\n{gossip_lines}"
```

Альтернативно — отдельным аргументом в `decide(... social_signal=...)`. Это аддитивная правка ~10–30 строк, не пересборка архитектуры.

### 1.3. Что доступно по данным для LLM-gossip

К моменту обработки агента N+1 в `run_llm_spike.py` / `run_llm_agents.py` доступны:

- `slot_loads[sid][hall]` — счётчик по залу (N выбравших среди первых N агентов до текущего);
- эквивалент `local_choice_count[talk_id]` через `talks_by_hall` mapping (на текущих программах эквивалентно `slot_loads[sid][hall]`, поскольку 1 talk на 1 hall в slot);
- эмбединги персон → можно вычислить cohort-сходство для L3.

Никакой новой инфраструктуры данных не требуется — те же сущности, что для параметрического gossip.

## 2. Варианты L0–L5+

Для каждого варианта: суть, реализация, привязка к `w_gossip` параметрического симулятора, что значит `w_gossip = 0`, проверки, плюсы / минусы / риски, реализуемость в срок.

### L0. LLM без gossip (status quo / rejected по решению пользователя)

- **Суть:** LLM-симулятор остаётся в виде `LLMAgent` V3 (profile + history + recommendation), gossip-сигнал не передаётся. Параметрический имеет `w_gossip ∈ [0, 0.7]`, LLM имеет «имплицитный» gossip-сигнал через естественный приор языковой модели.
- **Что менять в коде:** ничего. Status quo.
- **Связь с параметрическим gossip:** **никакой.** LHS-точки с `w_gossip > 0` в параметрическом сравниваются с LLM-точкой без `w_gossip`-варьирования. Cross-validation сравнивает разные модели отклика.
- **Управление через `w_gossip`:** не имеет аналога. LLM-симулятор не реагирует на `w_gossip` параметрического симулятора.
- **Что значит `w_gossip = 0`:** в LLM — всегда «как сейчас», независимо от параметрического значения.
- **Как проверить:** Spearman ранжирования политик параметрического и LLM на 12 точках LHS, как в spike_llm_simulator §11 acceptance V.
- **Плюсы:** zero-cost, статус-кво.
- **Минусы:** **отвергнуто пользователем**; LHS-cross-validation между двумя симуляторами «не сходится» по составу каналов; в защите трудно объяснить «параметрический имеет три канала, LLM два».
- **Риски для защиты:** оппонент задаёт «почему LLM-симулятор без gossip, если PROJECT_DESIGN §7 требует трёхкомпонентную модель?» — нужен заранее заготовленный ответ.
- **Реализуемость:** trivial.
- **Решение:** **rejected по решению пользователя 2026-05-07.**

### L1. Простая агрегированная prompt-вставка

- **Суть:** В `USER_PROMPT_TEMPLATE` добавляется одна-две строки текста с агрегированным сигналом по слоту: «В этом слоте больше всего людей сейчас идут на доклад X». Сигнал считается на стороне `run_llm_spike.py` из `slot_loads[sid]` (или `local_choice_count`).
- **Что менять в коде:** ~20 строк в `llm_agent.py` (новый параметр `decide(... gossip_text=None)` + блок в template) + 5–10 строк в `run_llm_spike.py` (формирование `gossip_text` из `slot_loads`).
- **Связь с параметрическим gossip:** **слабая.** Сигнал словесный («больше всего», «многие»), без явных count/rate. Эмпирически связан с `local_choice_count`, но LLM сам решает, как реагировать.
- **Управление через `w_gossip`:** через **системный промпт**: при `w_gossip = 0` — секции нет; при `w_gossip ∈ (0, 0.4)` — нейтральная фраза («заметная часть участников выбрала X»); при `w_gossip ≥ 0.4` — акцентированная («большинство уже идут на X»). Это **дискретная аппроксимация** continuous `w_gossip`.
- **Что значит `w_gossip = 0`:** gossip-блок не вставляется в промпт; промпт идентичен текущему `LLMAgent` V3.
- **Как проверить:** A/B на toy_microconf_2slot — `w_gossip = 0` vs `w_gossip = 0.5` (high-формулировка). Распределение `chosen` должно различаться. Diagnostic: процент агентов, явно ссылающихся на «другие выбрали» в `reason`.
- **Плюсы:** простая реализация, прямая семантика, не требует новой инфраструктуры. Сохраняет паритет с параметрическим в смысле «оба видят что-то про популярность».
- **Минусы:** дискретная аппроксимация `w_gossip` (3 уровня вместо continuous); LLM может игнорировать сигнал; калибровка интенсивности словесная, не числовая. Эмпирически — нельзя доказать, что L1 даёт «такую же силу» gossip-эффекта, что параметрический V5 при том же `w_gossip`.
- **Риски для защиты:** оппонент: «почему вы используете 3 формулировки, а параметрический — continuous? Это не одинаковые модели». Ответ: «Это не должно быть одинаковым по форме — это разные операционализации одного компонента; согласованность проверяется на уровне ранжирования политик, не на уровне силы сигнала».
- **Реализуемость:** малая, ~30 строк новой реализации; ~2 часа работы. Ложится в этап K без рефакторинга.

### L2. Тот же `log(1 + count_t) / log(1 + N_users)`, переведённый в естественный язык с реальными числами

- **Суть:** В промпт передаётся **точная статистика** per talk: «Доклад t_s0_ml: 7 из 10 участников уже выбрали; доклад t_s0_ios: 2 из 10». LLM получает реальные числа `count_t` и `N_users`, идентичные параметрическому. Системный промпт инструктирует «учитывать популярность как один из факторов».
- **Что менять в коде:** ~25 строк в `llm_agent.py` + 10 строк в `run_llm_spike.py` (формирование таблицы count_t). Новый блок в `USER_PROMPT_TEMPLATE`:
  ```
  Текущая популярность вариантов в этом слоте (по выбору других участников):
    - {talk1.title}: {count_1}/{N_users}
    - {talk2.title}: {count_2}/{N_users}
  ```
- **Связь с параметрическим gossip:** **сильная.** Оба симулятора используют **одну и ту же семантику** gossip-сигнала — количество уже выбравших доклад участников в текущем слоте (`count_t` и `N_users`). Конкретные значения `count_t` эндогенны внутри каждого симулятора и могут различаться (разные модели выбора → разные траектории `count_t` на одной и той же конференции и seed). Различие — и в источнике значений, и в способе взвешивания: параметрический использует точную `log(1+count)/log(1+N)` через `w_gossip` в utility; LLM получает текст с эндогенным `count_t` своей траектории и взвешивает «как сочтёт нужным».
- **Управление через `w_gossip`:** через **системный промпт-уровни**: при `w_gossip = 0` — блок выключен полностью (не передаётся); при `w_gossip ∈ (0, 0.4)` — «учитывай популярность как умеренный фактор»; при `w_gossip ≥ 0.4` — «учитывай популярность как сильный фактор». Опционально: **точное число `w_gossip`** в системный промпт «социальное влияние интенсивностью {w_gossip:.2f}», чтобы LLM сам интерпретировал. Это даёт continuous параметризацию, но её действенность зависит от способности LLM понимать число.
- **Что значит `w_gossip = 0`:** gossip-блок и инструкция не вставляются в промпт; промпт идентичен текущему `LLMAgent` V3 (соответствует L1 при `w_gossip = 0`).
- **Как проверить:** (1) A/B на toy: `w_gossip = 0` vs `w_gossip = 0.5` распределение `chosen` различается; (2) **сопоставление сигналов**: для каждого решения проверить, что `count_t` в промпте совпадает с `local_choice_count[t]` параметрического в той же конфигурации (паритет данных); (3) sensitivity по `w_gossip ∈ {0, 0.3, 0.7}`: rust в концентрации выбора (Gini) монотонно неубывающая.
- **Плюсы:** **прямой паритет данных с параметрическим** — оба симулятора видят одни и те же числа `count_t`. Легко обосновать в защите как «общий социальный сигнал, разные модели его интерпретации». Continuous параметризация через системный промпт.
- **Минусы:** LLM может игнорировать или гиперреагировать на числа — нет гарантии, что эффект будет монотонным по `w_gossip`. Калибровка через системный промпт остаётся «рукодельной».
- **Риски для защиты:** меньше, чем у L1. Оппонент может спросить «почему LLM реагирует именно с такой интенсивностью на `w_gossip = 0.5`?» — ответ: «Эмпирическая калибровка через A/B на toy-cases (этап L)».
- **Реализуемость:** малая, ~35 строк, ~3 часа работы.

### L3. OASIS-inspired controlled social signal layer (cohort-level)

- **Суть:** Социальный сигнал передаётся на уровне **когорты** — группа участников с близким эмбедингом профиля. Реализация по образцу OASIS (S12 spike_gossip), но без полного 5-компонентного OASIS (Environment Server / RecSys / Agent Module / Time Engine / Scalable Inferencer) — берём только controlled social signal layer.
  
  Алгоритм:
  1. Pre-compute K-means кластеры на эмбедингах персон (`run_llm_spike.py` уже делает это для отбора 10 разнообразных персон через KMeans-k=10).
  2. Для каждого слота поддерживается `cohort_choice_count[cluster_id][talk_id]`.
  3. В промпт передаётся «среди людей с похожими интересами {N_cohort} из них уже выбрали {talk_t}: {cohort_count}». Текущий пользователь видит сигнал только своей когорты.
- **Что менять в коде:** ~70 строк. Новые: `cohort_assignment` (pre-compute), `cohort_choice_count: Dict[int, Dict[str, int]]`, обновление per slot, формирование промпт-блока. Возможный новый параметр `n_cohorts` в SimConfig / CLI.
- **Связь с параметрическим gossip:** **средняя.** Параметрический V5 — global aggregate, L3 — cohort-aggregate. Это **разный сигнал**, не паритет данных. Параметрический видит «7 из 10 выбрали t», LLM видит «3 из 4 в моей когорте выбрали t».
- **Управление через `w_gossip`:** аналогично L2 — через системный промпт + опционально число.
- **Что значит `w_gossip = 0`:** cohort-блок не вставляется.
- **Как проверить:** более сложная проверка. (1) A/B на toy сложнее, потому что нужно ≥2 когорты ⇒ ≥10 агентов; (2) cohort-stability: при разных seed'ах кластеризации структура социального сигнала меняется — это новый источник шума, требует усреднения.
- **Плюсы:** научно богаче (S20 social proof — similarity усиливает effect; S4 Salganik 2006 cohort-cascade сильнее aggregate). Эмпирически правдоподобнее (Cialdini).
- **Минусы:** **разрывает паритет данных** с параметрическим V5 — два симулятора видят разные сигналы. Если хочется сохранить паритет, **параметрический нужно тоже расширить до cohort-V6** (и тогда раздувается параметрический), или **оба расходятся**, и cross-validation ослабевает. Сложнее реализация. Параметризация cohort_id, K-means seed добавляет осей шума.
- **Риски для защиты:** оппонент: «вы используете cohort-сигнал в LLM, а в параметрическом — global? Это разные модели». Ответ: «Это **разные** механизмы социального заражения — global social proof и cohort social proof; результат cross-validation интерпретируется как робастность относительно формы социального сигнала, не относительно его силы».
- **Реализуемость:** средняя, ~7–10 часов работы. Не вписывается в одну сессию этапа K, требует отдельного дробления.

### L4. Inter-agent chat / social graph / полная social simulation

- **Суть:** Полный OASIS / RecAgent (S12, S14) — каждый агент пишет / читает сообщения, есть social graph, MQTT-уровень коммуникации.
- **Что менять в коде:** новый модуль 200–500 строк. Refactor `LLMAgent.decide` — двухпроходная обработка слота (фаза «обмен сообщениями» + фаза «выбор»).
- **Связь с параметрическим gossip:** **рассыпается**. Параметрический становится грубой аппроксимацией для проверки качества полной social simulation, но не наоборот.
- **Управление через `w_gossip`:** не имеет естественной интерпретации. Параметризация — через число шагов диалога, длину сообщений, охват графа.
- **Минусы:** **прямой стоп-лист §5 PROJECT_STATUS** («inter-slot chat / social graph как реализованный метод — был прототип, не основной результат»). Дорого по $ (×N LLM-вызовов на N агентов в каждом слоте). Сильно сдвигает работу в направление «полноценной social ABM-среды», что противоречит постановке ВКР как DSS.
- **Реализуемость:** очень большая, не реализуема в timeline до 13.05.
- **Решение:** **rejected** (стоп-лист + срок).

### L5. Pass-through scalar gossip-score из параметрического

- **Суть:** Параметрический gossip-расчёт делается в `run_llm_spike.py` (по той же формуле V5). Полученный per-talk score `g_t = log(1 + count_t) / log(1 + N_users)` передаётся в LLM-промпт как **числовой ярлык** на каждый вариант:
  ```
  Доклад t_s0_ml: social_score = 0.83 (по шкале 0..1 — насколько уже выбран другими)
  ```
  Системный промпт инструктирует «учитывай social_score как один из факторов».
- **Что менять в коде:** ~20 строк в `llm_agent.py` + 5 строк в `run_llm_spike.py`. Минимум.
- **Связь с параметрическим gossip:** **очень высокая.** Не просто паритет данных, но и **паритет вычисленной величины** (тот же score по той же формуле). LLM-симулятор использует тот же сигнал, что параметрический; различие только в способе взвешивания (параметрический — через `w_gossip` в utility-MNL; LLM — через инструкцию в системном промпте).
- **Управление через `w_gossip`:** более прямое, чем в L1/L2. При `w_gossip = 0` блок не вставляется. При `w_gossip > 0` блок вставляется + инструкция «учитывай этот score сильно/средне». Опционально: continuous через «вес `w_gossip`» в инструкции.
- **Что значит `w_gossip = 0`:** социальный score не передаётся; промпт идентичен текущему `LLMAgent` V3.
- **Как проверить:** прямой паритет данных — сравнить `social_score` в LLM-промпте с `gossip(t, L_t)` параметрического для каждого decision step. Должны совпадать **байт-в-байт**. Sensitivity — как у L2.
- **Плюсы:** **наивысший паритет** между двумя симуляторами. Cleanest для cross-validation. Минимум кода.
- **Минусы:** **сильно ослабляет «независимость» двух симуляторов.** Если LLM получает уже посчитанный gossip-score, она становится наполовину параметрической: реагирует на тот же inputs через interpretation, но не на «своей» интерпретации сигнала. Cross-validation на 12 точках LHS становится близкой к «один и тот же сигнал, две разных response model» — это уже **не** «два независимых источника отклика» в строгом смысле PROJECT_DESIGN §7.
- **Риски для защиты:** оппонент: «вы передаёте уже вычисленный социальный score в LLM — это ослабляет независимость двух симуляторов. Где вторая независимая модель?». Ответ требует чёткой оговорки в Limitations: «после amendment паритет данных приоритетнее независимости интерпретации; cross-validation проверяет согласованность ranking-политик при разных response-моделях, не разных сигналах».
- **Реализуемость:** малая, ~25 строк, ~2 часа.

### L6 (новый, дополнительный). Гибрид L2 + L5: естественные числа + явный score

- **Суть:** В промпт передаются и сырые числа `count_t`/`N_users`, и нормализованный `social_score`. Системный промпт инструктирует «учитывай это как social signal».
- **Связь с параметрическим:** очень высокая (L5-уровень) + дополнительная избыточная читаемость (L2-уровень).
- **Минусы:** перегруженный промпт; LLM может «дважды учесть» сигнал.
- **Решение:** менее предпочтителен, чем L2 или L5 по отдельности.

## 4. Согласованность с accepted Q-G «capacity не в LLM-промпте»

Q-G accepted в `spike_llm_simulator.md`: «Capacity в промпт LLM-агента не передавать». Пользователь сейчас просит передавать gossip в промпт. Кажется противоречием.

**Это не противоречие, потому что capacity и gossip — разные семантические сущности:**

- **Capacity** = структурное ограничение (статичная вместимость зала) + динамическая загрузка. Если LLM-агент видит `load_frac`, он начинает «играть» в capacity-aware политику изнутри — это **дублирует** канал политики П3 и нарушает PROJECT_DESIGN §9 («политика влияет только через `w_rec`»).
- **Gossip** = чисто динамический social signal по выбору других. Не структурное свойство среды, а production эмерджентного coordination. PROJECT_DESIGN §7 явно делает gossip **компонентом utility агента**, не свойством политики.

Поэтому: **accepted Q-G сохраняется** (capacity не в промпте); **amendment** (gossip в промпте) **не нарушает** Q-G — это другая семантическая сущность.

Тонкий момент: при использовании per-talk `count_t` (рекомендованный сигнал для V5), на текущих программах `count_t == load[hall(t)]` тождественно. То есть LLM-агент при L2/L5 видит цифру, которая численно совпадает с `load[hall]`. Формально это всё ещё не capacity (нет ссылки на `capacity[hall]` или `load_frac`), а социальный count. Но **семантически** агент может «угадать», что это связано с заполненностью зала. Чтобы не дать ему этого — формулировка в промпте должна быть явно социальной: «сколько людей выбрало этот доклад», а не «сколько людей в зале».

**Operational decision:** в L1/L2/L5 сигнал **формулируется** как «выбор других участников» (по talk_id), не как «загрузка зала» (по hall_id). Это сохраняет **семантическое разделение** между gossip-каналом и capacity-каналом, даже если данные численно идентичны на наших программах.

## 5. Independence vs comparability tension

Главный архитектурный конфликт после amendment: **независимость двух симуляторов** (PROJECT_DESIGN §7) vs **сопоставимость на 12 точках LHS** (PROJECT_DESIGN §11).

- **L0 (отвергнут):** максимум независимости, минимум сопоставимости.
- **L4 (отвергнут):** другая модель отклика, формально независима, но не сопоставима.
- **L1, L2:** баланс. Сигнал передаётся, но LLM сам интерпретирует. Параметризация дискретная (L1) или текстово-continuous (L2). Независимость частично сохраняется через interpretation freedom.
- **L3 cohort:** баланс другой формы. LLM получает другой сигнал (cohort-aggregate), параметрический — global. Симуляторы используют разные операционализации gossip.
- **L5 pass-through:** максимум сопоставимости, минимум независимости. LLM становится «параметрической моделью отклика на тот же сигнал».

Канон литературы:

- **S15 Larooij & Törnberg 2025** (`spike_llm_simulator.md` S11) — believability ≠ validity; cross-validation через **distribution-match**, не через repetition. Это поддерживает L1/L2 (LLM реагирует «по-своему», но согласие распределений на ranking-уровне).
- **S6 Chaney 2018**, **S7 Mansoury 2020** — feedback loop проявляется в любом случае; для атрибуции нужен control `w_gossip = 0`.
- **S12 OASIS** — controlled signal layer без полной social simulation работает (поддерживает L3 как «OASIS-inspired», но не «full OASIS»).

## 6. Recommendation

**Рекомендованный вариант для этапа K (LLM-сторона):** **L2 — реальные числа `count_t` / `N_users` в промпте + дискретно-continuous параметризация через системный промпт по уровням `w_gossip`**.

**Содержательно:**

```python
# В run_llm_spike.py (новый вспомогательный код):
def render_gossip_block(slot_loads_so_far: Dict[int, int],
                        slot_talks: List[Dict],
                        n_users: int) -> str:
    if not slot_loads_so_far:
        return ""
    lines = []
    for t in slot_talks:
        count = slot_loads_so_far.get(t["hall"], 0)
        lines.append(f"  - {t['title'][:60]}: {count} из {n_users}")
    return "\nЧто уже выбрали другие участники в этом слоте:\n" + "\n".join(lines)

def render_gossip_instruction(w_gossip: float) -> str:
    if w_gossip == 0.0:
        return ""
    if w_gossip < 0.4:
        return "\nУчитывай выбор других участников как умеренный фактор."
    return "\nУчитывай выбор других участников как сильный фактор."
```

В `LLMAgent.decide` дополняется параметром `gossip_text=None`; при `gossip_text != None` вставляется в `USER_PROMPT_TEMPLATE` после `rec_block`.

Системный промпт расширяется одной строкой при `w_gossip > 0`.

**Обоснование:**

1. **Паритет семантики сигнала с параметрическим:** оба симулятора используют **одну и ту же семантику** gossip-сигнала — количество уже выбравших доклад участников в текущем слоте. `count_t` и `N_users` — те же по семантике величины, что считает параметрический V5; конкретные значения эндогенны внутри каждого симулятора и могут различаться из-за разных моделей выбора.
2. **Независимость интерпретации сохраняется:** LLM сам решает, как взвесить эндогенный для него `count_t`, не получает готовый score (как в L5). Это удовлетворяет PROJECT_DESIGN §7 «второй независимый источник отклика».
3. **Минимальная реализация:** ~35 строк, без новых модулей.
4. **Не нарушает Q-G accepted:** capacity не передаётся; gossip формулируется как «выбор других», не «загрузка зала».
5. **Не превращает работу в social ABM:** нет inter-agent chat, нет social graph, нет MQTT.
6. **Параметризация через `w_gossip`:** дискретные уровни 0 / med / high. Это аппроксимация, но достаточная для проверки на toy и согласованности на 12 точках LHS.
7. **Проверяемость:** A/B на toy, sensitivity по `w_gossip`. Численного сравнения `count_t` в LLM-промпте с `count_t` параметрического в той же конфигурации **не делаем** — траектории эндогенны и в общем случае различаются.

**Engineering choice оговорка (Q-J11 accepted 2026-05-07):** L2 — инженерное решение, мотивированное литературой по LLM-симуляторам с социальным сигналом (S12 OASIS — log-scale popularity score; S13 Agent4Rec; S20 social proof; S6, S7 recsys feedback loops). Это **не прямое воспроизведение** OASIS-architecture (которая включает Environment Server, RecSys, Agent Module, Time Engine, Scalable Inferencer и `21 действие`); из OASIS заимствуется только идея controlled aggregated social signal в text-form промпте. Полноценная social-ABM-среда отвергнута (см. L4) как несовместимая со сроками и стоп-листом §5 PROJECT_STATUS.

**Что отклоняется и почему:**

- **L0** — пользователь явно отверг.
- **L1** — слабее L2 по паритету данных без выгоды (тот же объём кода).
- **L3 cohort** — научно богаче, но разрывает паритет данных и втягивает в новую ось эксперимента (cohort-параметры). Откладывается на «later» (раздел 10 уточнение в основном `spike_gossip.md`).
- **L4 full OASIS** — стоп-лист §5 + срок.
- **L5 pass-through scalar** — paritetно идеально, но превращает LLM-симулятор в «вторую model отклика на тот же сигнал», ослабляя независимость. Если в L удастся проверить, что L2 даёт нестабильный sensitivity по `w_gossip`, можно перейти к L5 как fallback. Но не в первую версию.
- **L6** — избыточен.

**Альтернатива:** если пользователь предпочитает максимальную сопоставимость в ущерб независимости — **L5** (pass-through scalar). Я выношу этот выбор в Q-J7-revised.
