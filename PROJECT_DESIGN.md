## 1. Тема

«Разработка интеллектуальной системы поддержки формирования программы конференции».

Тема утверждена; переформулировке не подлежит. Содержательная привязка задаётся подзаголовком, целью и постановкой задачи.

## 2. Краткая суть

Объект автоматизации — работа организатора по составлению и оценке программы конференции. На стадии планирования у организатора есть только описание самой конференции: список временных слотов, набор параллельных докладов в каждом слоте, привязка докладов к залам, вместимости залов, тексты докладов, состав спикеров. Сведений о реальной аудитории, её предпочтениях и фактической посещаемости на этом этапе нет и не может быть.

Поведение аудитории система моделирует. Модель собирается из четырёх компонентов: оценка популярности доклада, модель индивидуального выбора между параллельными докладами, модель социального заражения (gossip), модель реакции аудитории на рекомендацию. На этих моделях строится численный эксперимент: программу конференции прогоняют через серию параметрических конфигураций. Результат эксперимента есть количественная характеристика ожидаемой загрузки залов, плюс сравнение политик рекомендаций и сравнение модификаций программы.

Главная функция системы — превратить неопределённость спроса на параллельные сессии в структурированный сравнительный отчёт. На основании этого отчёта организатор принимает решение об окончательной структуре программы. По жанру разрабатываемая система относится к классу систем поддержки принятия решений (СППР, Decision Support System, DSS).

### Контекст разработки

Работа ведётся в составе продуктовой экосистемы для конференций сообщества JUG. Существующая часть экосистемы — телеграм-клиент, в котором участник в формате попарного отбора (свайп) формирует персональное расписание докладов, получает напоминания и проставляет оценки прослушанным сессиям. Клиент построен на стеке backend + frontend + векторное хранилище Qdrant с поиском по семантической близости эмбеддингов докладов; технически развёртывался в production-среде на конференциях сообщества. Разрабатываемая в настоящей работе СППР проектируется как аналитический слой этой же продуктовой экосистемы, используемый организатором на этапе подготовки программы.

## 3. Цель работы

Разработать программную систему, выполняющую численный сравнительный анализ загрузки залов, эффектов политик рекомендаций и эффектов локальных модификаций программы. Анализ опирается на синтетическую модель аудитории и на параметризуемый набор предположений о её поведении.

## 4. Объект и предмет

Объектом исследования является процесс формирования программы конференции с параллельными сессиями и ограниченной вместимостью залов в условиях, когда данные о фактической аудитории недоступны.

Предмет исследования — численные методы оценки распределения аудитории по залам, методы моделирования индивидуального выбора между параллельными докладами, методы оценки эффекта политик рекомендаций и локальных модификаций программы на распределение нагрузки, а также методы анализа чувствительности результата к параметрам моделирования.

## 5. Постановка задачи

### Дано

Произвольная программа конференции $P = \{S, T, H, A, c, M\}$, где $S$ есть множество временных слотов, $T$ — множество докладов, $H$ — множество залов, $A: T \to H \times S$ — отображение «доклад — зал, слот», $c: H \to \mathbb{N}$ — функция вместимости залов, $M$ — метаданные (тексты докладов, спикеры, тематические признаки).

Параметризуемые модельные объекты:

1. Модель популярности $\rho: T \to \mathbb{R}_{\geq 0}$ — априорная привлекательность доклада. Оценивается по тексту доклада, по метаданным спикеров и (в одном из вариантов реализации) языковой моделью.
2. Модель синтетической аудитории $U_\theta$ — порождающий процесс, который выдаёт множество индивидуальных профилей участников.
3. Модель индивидуального выбора $\sigma$ — функция, по профилю участника, доступным в слоте докладам, рекомендации и текущей загрузке залов задающая распределение вероятностей выбора.
4. Модель социального заражения $G$ — динамическое усиление выбора в зависимости от того, что уже выбрала когорта.

Кроме того, заданы множество политик рекомендаций $\Pi$ и оператор локальной модификации программы $\Phi$. Оператор $\Phi$ выдаёт набор допустимых вариантов программы $\{P_0, P_1, \dots\}$ при двух ограничениях: состав докладов фиксирован, конфликты спикеров не допускаются. Множество параметрических конфигураций $\Xi$ есть пространство всевозможных комбинаций значений по всем параметрическим осям: вместимость, параметры модели популярности, веса модели поведения участника, размер аудитории, вариант программы.

### Требуется разработать

Систему, которая для произвольной программы $P$, конфигурации $\xi \in \Xi$ и политики $\pi \in \Pi$ выполняет следующее:

1. Генерирует синтетическую аудиторию по параметрам конфигурации.
2. Оценивает популярность докладов выбранной моделью.
3. Воспроизводит индивидуальный выбор каждого участника во всех слотах при заданной политике, заданной модели поведения и заданном варианте программы.
4. Регистрирует распределение посещений по парам «слот × зал».
5. Считает показатели загрузки и риска превышения вместимости.
6. Повторяет шаги 1–5 для всех конфигураций $\xi \in \Xi^*$ и политик $\pi \in \Pi$. Здесь $\Xi^*$ — выборка из $\Xi$, формируемая по плану эксперимента (см. раздел 11).
7. Собирает сравнительный отчёт.

### Выходные данные

На выходе система предъявляет распределение ожидаемой загрузки по парам «слот × зал» для каждой пары $(\xi, \pi)$, список троек «слот × зал × доклад», в которых ожидаемая загрузка превышает вместимость (далее — горячие точки программы), таблицу показателей по комбинациям «политика × конфигурация», перечень локальных модификаций программы со снижением риска относительно исходной программы, характеристики чувствительности результата к параметрам конфигурации и раздел допущений.

## 6. Главный пользователь и форма результата

Целевой пользователь системы — организатор программы конференции.

Результат предъявляется в виде сравнительного отчёта. Отчёт содержит карту ожидаемой загрузки залов, список горячих точек программы, сравнительные таблицы политик рекомендаций, перечень локальных модификаций программы с оценкой их эффекта, результаты анализа чувствительности к параметрам и раздел допущений.

Решение об изменении программы или о выборе политики принимает организатор. Функция системы — численная и сравнительная; роли предписывающего инструмента у неё нет.

## 7. Состав системы

Система собрана из девяти модулей.

### Модуль программы конференции

Хранит внутреннее представление расписания: слоты, доклады, залы, вместимости, привязки докладов, тексты докладов, спикеров. Программа принимается в произвольном формате при условии, что она удовлетворяет спецификации входных данных. От этого модуля зависят все остальные.

### Модуль модели популярности

Оценивает априорную привлекательность доклада $\rho: T \to \mathbb{R}_{\geq 0}$. Сигналом служит семантическая близость темы доклада к тематике конференции, известность спикера, оценка языковой модели. Конкретный источник выбирается параметром конфигурации; модуль содержит несколько взаимозаменяемых реализаций.

### Модуль синтетической аудитории

Порождающий процесс, выдающий множество профилей участников. Параметры процесса: размер аудитории и распределение тематических интересов.

### Модуль модели поведения участника

Задаёт индивидуальный выбор между параллельными докладами. В функцию полезности входят три компонента, каждый управляется отдельным весом. Компонент релевантности отражает близость доклада к профилю участника, $w_{rel}$. Компонент следования рекомендации отражает влияние показанной политикой подсказки, $w_{rec}$. Компонент социального заражения (gossip) описывает динамическое усиление выбора по тому, что уже выбрала когорта, $w_{gossip}$. Кроме весов, у модели есть параметр стохастичности.

### Модуль политик рекомендаций

Семейство политик в едином интерфейсе. Политика получает профиль участника, доступные в слоте доклады и текущую оценку загрузки залов, возвращает упорядоченный список из K докладов. На итоговое распределение посещений политика влияет только через компонент $w_{rec}$ функции полезности.

### Модуль локальных модификаций программы

Реализует оператор $\Phi$. Оператор обменивает пары докладов между параллельными слотами одного дня; состав докладов фиксирован, конфликты спикеров не допускаются. Полученные варианты программы поступают в симулятор так же, как исходная программа.

### Модуль симуляции

Существует в двух независимых вариантах. Параметрический вариант реализует модель индивидуального выбора замкнутой формы — мультиномиальный логит. Агентский вариант построен на языковой модели: каждому участнику соответствует отдельный экземпляр языковой модели с профилем-персоной и историей выборов в текущем слоте. Агентский симулятор играет роль второго независимого источника отклика и нужен для перекрёстной проверки выводов параметрического.

### Модуль показателей

Считает количественные характеристики результата. Показатели сгруппированы по смыслу: загрузка и её распределение, риск превышения вместимости, потеря релевантности рекомендаций, устойчивость политики при варьировании конфигурации, эффект модификации программы относительно исходной.

### Модуль сравнительного отчёта

Собирает результаты в форму, понятную организатору: карта загрузки, список горячих точек, сравнительные таблицы политик, перечень модификаций программы с количественной оценкой эффекта, графики чувствительности, раздел допущений.

## 8. Параметрические оси численного эксперимента

Под конфигурацией далее понимается точка в пространстве параметрических осей. Эксперимент состоит в проходе по выборке таких точек.

Ось 1 — вместимость залов. Управляется множителем относительно номинальной вместимости и режимом распределения мест между залами одного слота.

Ось 2 — модель популярности. Источник оценки $\rho(T)$: косинусная близость к тематике конференции, известность спикера, оценка языковой модели или их комбинация.

Ось 3 — параметры модели поведения участника. Веса $(w_{rel}, w_{rec}, w_{gossip})$ и параметр стохастичности.

Ось 4 — размер и состав аудитории. Численность; распределение тематических интересов.

Ось 5 — вариант программы. Исходная программа $P_0$ или одна из модификаций $P_k$, полученных оператором $\Phi$ при фиксированном бюджете локальных перестановок. Реализуется как индекс варианта в множестве $\{P_0, P_1, \dots, P_k\}$.

Ось 6 — случайность. Несколько повторов на каждую конфигурацию для усреднения.

Политика рекомендаций является отдельной осью; внутри каждой точки выборки она перебирается полностью.

## 9. Состав политик рекомендаций

Под политикой далее понимается алгоритм формирования упорядоченного списка из K докладов, предъявляемого участнику в слоте. На итоговое распределение посещений политика влияет только через компонент $w_{rec}$ функции полезности участника. При $w_{rec} \to 0$ политики неразличимы по итогу; при $w_{rec} \to 1$ выбор политики определяет итог полностью.

В сравнении участвуют четыре политики.

П1 — контрольная. Рекомендация участнику не предъявляется; компонент $w_{rec}$ из функции полезности исключён. Эта политика играет роль точки отсчёта.

П2 — по релевантности (cosine). Top-K строится ранжированием докладов по близости к профилю участника. Сигналы загрузки и популяции при этом не учитываются.

П3 — с учётом загрузки (capacity-aware). К скорингу добавляется компонент, монотонно убывающий по текущей загрузке зала.

П4 — на основе языковой модели (LLM-ranker). Профиль участника и список кандидатов передаются языковой модели, и она возвращает ранжирование.

Расширения — диверсификация рекомендаций (MMR) и внешний сигнал популярности (gossip-вход в скоринг политики) — подключены как параметрические модификаторы; в основном сравнении они не выделяются в виде отдельных политик.

## 10. Показатели

Показатели сгруппированы по смыслу. Группы 2 и 4 центральны: первая операционализирует понятие риска перегрузки, вторая — устойчивость политики при варьировании конфигурации.

### Группа 1. Загрузка и распределение

Средняя загрузка зала в слоте; дисперсия загрузки залов внутри слота; коэффициент неравенства загрузки между залами слота.

### Группа 2. Риск превышения вместимости

Среднее относительное превышение вместимости по тем парам «слот × зал», в которых превышение зафиксировано. Доля слотов, в которых хотя бы один зал превышает вместимость. Мощность множества горячих точек.

### Группа 3. Релевантность рекомендаций

Средняя релевантность фактически выбранного доклада профилю участника. Потеря релевантности относительно политики П2 при тех же параметрах конфигурации.

### Группа 4. Устойчивость политики

Доля точек выборки, в которых одна политика лучше другой по показателю риска (попарное сравнение). Размах значений показателя риска при варьировании одной оси конфигурации. Сожаление политики относительно лучшей в каждой точке.

### Группа 5. Эффект модификации программы

Относительное снижение показателей риска от перестановки относительно исходной программы при фиксированной политике и фиксированной конфигурации. Доля конфигураций, в которых модификация даёт значимое снижение риска при выбранной политике.

### Группа 6. Совместные характеристики

Двумерное распределение пар «риск × релевантность» по точкам и политикам.

## 11. Экспериментальный протокол

Эксперимент построен на латинском гиперкубе по всем параметрическим осям, кроме политики; политика перебирается полным перебором внутри каждой точки гиперкуба. Такой план соответствует канону планирования имитационных экспериментов (Kleijnen 2005).

### Параметрический симулятор

Гиперкуб строится из 50 точек по осям конфигурации (вместимость, модель популярности, параметры модели поведения, размер аудитории, вариант программы). Внутри каждой точки перебираются все четыре политики, и для каждой пары $(\xi, \pi)$ симулятор запускается с тремя различными случайными зёрнами. Суммарный объём — порядка 600 прогонов; временной бюджет — минуты.

### LLM-симулятор

Из тех же 50 точек отбираются 12, покрывающие крайние и центральные значения каждой оси. Внутри каждой точки также перебираются все четыре политики. Случайное зерно одно: внутри одной точки все четыре политики прогоняются на одной и той же синтетической аудитории — это известный приём общих случайных чисел (common random numbers), снимающий шум сравнения политик. Суммарный объём — порядка 48 прогонов; временной бюджет — часы.

### Анализ результатов

Для каждой пары $(\xi, \pi)$ считаются среднее и разброс показателей раздела 10.

Сравнение политик идёт попарно внутри каждой точки гиперкуба: «доля точек, в которых политика A лучше политики B по показателю $m$». Усреднение показателя по точкам не используется, поскольку оно скрывает сценарную чувствительность.

Сравнение модификаций программы идёт аналогично — попарно между $P_0$ и $P_k$ при фиксированной политике и фиксированных остальных параметрах.

Чувствительность результата исследуется по каждой оси отдельно: фиксируются все параметры, кроме одного, и наблюдается изменение показателя риска вдоль этой оси.

Согласованность двух симуляторов проверяется на 12 общих точках. Сопоставляются ранжирования политик в LLM-эксперименте и в параметрическом эксперименте. Качественное совпадение ранжирования трактуется как взаимная проверка двух независимых источников отклика. Систематическое расхождение фиксируется и интерпретируется отдельно.

### Верификация модели в граничных условиях

До содержательного анализа модель должна показать четыре свойства. При сильно избыточной вместимости показатели риска перегрузки стремятся к нулю независимо от политики. При уменьшении вместимости показатели риска монотонно растут. При $w_{rec} \to 0$ различие политик по показателям загрузки стремится к нулю. При $w_{rec} \to 1$ различие политик по показателям загрузки максимально. В литературе эту процедуру называют extreme condition testing (Sargent 2013).

Если хотя бы одно из четырёх свойств не выполняется, модель считается некорректной, и переход к содержательным выводам блокируется до устранения причины. Проверки выполняются автоматизированными интеграционными тестами на уровне модели в целом; с юнит-тестами кода они не пересекаются и решают другую задачу.

## 12. Ожидаемый результат

К защите ожидается следующее.

Программная система, реализующая модули раздела 7. Экспериментально полученные количественные характеристики загрузки залов и риска превышения вместимости для исследуемых программ при заданных конфигурациях. Сравнение политик рекомендаций по показателям загрузки, риска и релевантности с попарным сравнением и оценкой устойчивости. Перечень локальных модификаций программы с оценкой их эффекта на показатели риска при разных политиках и конфигурациях. Описание чувствительности результата к параметрам моделирования. Перечень горячих точек программы для исследуемых инстансов. Сравнительный отчёт в форме, пригодной для использования организатором при принятии решений о структуре программы.

## 13. Допущения и ограничения

Перечень фиксируется как часть условий задачи, а не как защитная оговорка.

Аудитория задаётся синтетическим порождающим процессом; согласованность синтетической аудитории с фактической предметом исследования не является.

Модель индивидуального выбора имеет параметрическую форму. Калибровка её параметров на внешних данных производится в одном из вариантов реализации, но в обязательный результат работы не входит.

Модель популярности докладов оценивается по признакам, доступным на этапе формирования программы. Соответствие этой оценки фактической популярности на конференции является внешним вопросом и в работе не рассматривается.

Реакция аудитории на рекомендации задаётся параметрически. Фактическая степень влияния рекомендации на реальной конференции трактуется как внешний параметр и варьируется как ось эксперимента.

Модификации программы рассматриваются как локальные перестановки в окрестности исходной программы. Полная задача оптимизации программы (целочисленное программирование на уровне всей решётки слотов и залов) к работе не относится; она разработана в существующей литературе по conference scheduling (Vangerven 2018, Pylyavskyy 2024, Bulhões 2022).

Численные значения показателей трактуются как сравнительные между политиками, конфигурациями и вариантами программы в рамках единой модели. Абсолютная интерпретация значений вне модели в работе не производится.

## 14. Связь с темой

Тема ВКР — «Разработка интеллектуальной системы поддержки формирования программы конференции». Каждый элемент темы получает в работе конкретное наполнение.

Интеллектуальная часть — это модели: оценка популярности доклада (в том числе оценка языковой моделью), индивидуальный выбор участника, социальное заражение, семейство политик рекомендаций (включая политику на основе языковой модели), оператор локальных модификаций программы.

Система — программная среда жанра СППР, которая собирает перечисленные модели и модули в воспроизводимый численный эксперимент с единым сравнительным отчётом и встраивается в существующую продуктовую экосистему JUG как аналитический слой для организатора.

Поддержку формирования программы организатор получает в виде этого отчёта. Из отчёта он узнаёт, где у программы горячие точки, какие политики устойчивее в каких сценариях, какие локальные перестановки снижают риск перегрузки. На основе отчёта организатор сам корректирует программу.

Программа конференции — единственный фактический вход системы. Все остальные сущности либо моделируются, либо задаются параметрами эксперимента.

## 16. Положения, выносимые на защиту

1. Предложена постановка задачи сценарной оценки загрузки залов при формировании программы конференции в условиях отсутствия данных о фактической посещаемости аудитории.
2. Разработана программная система поддержки принятия решений для организатора конференции, обеспечивающая сравнительный анализ политик рекомендаций и локальных модификаций программы.
3. Реализованы два независимых механизма моделирования отклика синтетической аудитории — параметрический и на основе больших языковых моделей, — позволяющие выполнять перекрёстную проверку выводов при сравнении политик рекомендаций.
4. Получены количественные оценки относительного эффекта политик рекомендаций и локальных модификаций программы по показателям риска перегрузки залов, баланса нагрузки и релевантности выбора.
5. Реализовано представление результатов сценарной оценки в виде сводных показателей и визуализаций, позволяющее сопоставлять загрузку залов, рисковые участки программы и эффекты рассматриваемых решений.
