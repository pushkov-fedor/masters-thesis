# Каркас Главы 3 (Реализация)

> Структурный план с описанием всех технических решений, реализованных в `experiments/`.
> Не финальный текст; это инвариантная заготовка, которую можно расширять до 8-12 страниц.

## 3.1. Архитектура решения и стек технологий

Решение реализовано как Python-пакет в каталоге `experiments/`:

```
experiments/
├── src/                  # переиспользуемая библиотека
│   ├── simulator.py      # параметрический симулятор
│   ├── metrics.py        # метрики оценки политик
│   └── policies/         # 7 рекомендательных политик
├── scripts/              # запускаемые сценарии
└── data/                 # программа конференции, эмбеддинги, персоны
```

Стек:
- Python 3.12 в локальном виртуальном окружении (`uv venv`).
- numpy, scipy, pandas — численные расчёты.
- sentence-transformers (paraphrase-multilingual-MiniLM-L12-v2, 384-dim) — эмбеддинги текстов на CPU без обращения к API.
- openai (асинхронный клиент) → OpenRouter API — генерация персон и LLM-ranker.
- matplotlib — графики.
- python-dotenv — управление API-ключом из `.env`.

Все вычисления локальны и воспроизводимы без серверной инфраструктуры.

## 3.2. Подготовка данных

### 3.2.1. Парсинг программы конференции

Источник — открытый Excel-файл `Mobius_Topics.xlsx`, содержащий лист `mobius_topics` с 40 докладами. Скрипт `scripts/build_dataset.py` извлекает поля id, title, abstract, category, hall, date, start_time, end_time и сохраняет нормализованную JSON-структуру в `data/conferences/mobius_2025_autumn.json` со схемой:

```json
{
  "conf_id": "mobius_2025_autumn",
  "talks": [{"id", "title", "speakers", "hall", "slot_id", "category", "abstract"}],
  "halls": [{"id", "capacity"}],
  "slots": [{"id", "datetime"}]
}
```

Вместимости залов (600/350/250 для залов 1/2/3) выставлены экспертно как разумные оценки порядков для конференц-зала Mobius (Cosmos Saint-Petersburg Pulkovskaya Hotel) — в открытых данных конкретные числа недоступны. Это допущение явно фиксируется как чувствительность результата.

Тайм-слоты автоматически извлекаются из (date, start_time) пар: 16 слотов, из которых 4 single-talk (открытие, закрытие, кейноуты) и 12 с тремя параллельными докладами в трёх разных залах.

### 3.2.2. Эмбеддинги докладов

Скрипт `scripts/compute_embeddings.py` преобразует каждый доклад в векторный эмбеддинг через модель `sentence-transformers/paraphrase-multilingual-MiniLM-L12-v2` (384 измерения). Текст для эмбеддинга формируется как `title — [category] — abstract`. Эмбеддинги нормализуются по L2 и сохраняются в `data/conferences/mobius_2025_autumn_embeddings.npz`.

### 3.2.3. Генерация синтетической пользовательской популяции

Скрипт `scripts/generate_personas.py` отправляет в OpenRouter 15 пакетных запросов к Claude Haiku 4.5 (модель `anthropic/claude-haiku-4.5`), каждый возвращает по 20 разнообразных персон в JSON-формате. Промпт направляет модель на генерацию реалистичных профилей участников Mobius: разные роли (iOS, Android, Flutter, KMP, React Native, QA, тимлид, продакт, DevRel, дизайнер), разные уровни (junior, middle, senior, lead/principal), разные размеры компаний (startup, midsize, large, enterprise).

Каждая персона имеет структуру:

```json
{
  "id": "u_001",
  "role": "iOS Senior Developer",
  "experience": "senior",
  "company_size": "large",
  "interests": ["SwiftUI", "performance"],
  "preferred_topics": ["Architecture", "Under the Hood"],
  "background": "развёрнутое текстовое описание профиля 2-4 предложения"
}
```

Стоимость генерации 300 персон — $0.29 (среднее 690 prompt + 3700 completion токенов на запрос).

Эмбеддинги профилей считаются локально через ту же модель multilingual-MiniLM по полю `background` и сохраняются в `data/personas/personas_embeddings.npz`.

### 3.2.4. Масштабирование популяции до 900 пользователей

Для достижения реалистичной нагрузки на залы (Mobius 2025 Autumn собирает порядка 800–1000 участников) 300 LLM-персон расширяются скриптом `scripts/expand_personas.py`: каждая персона реплицируется 3 раза с гауссовым шумом (σ=0.03) в эмбеддинге, после чего эмбеддинг возвращается на единичную сферу. Результат — 900 пользователей с локальной плотностью вокруг 300 центральных профилей. Идентификатор реплики — `<base_id>_r<i>`, что позволяет детерминированно сопоставлять реплики с базовой персоной для целей кэширования LLM-ranker.

## 3.3. Параметрический симулятор

`src/simulator.py` реализует пошаговую симуляцию пользовательского взаимодействия с программой конференции.

### 3.3.1. Постановка симуляции

Симулятор обходит слоты в порядке их следования. В каждом слоте перебираются пользователи в случайном порядке (детерминированном сидом). Для каждого пользователя:

1. Запрос политики `recommend(user, slot, conf, state)` возвращает список из K кандидатов.
2. Для каждого кандидата считается полезность $u_{ij} = r_{ij} - \lambda \cdot \mathrm{overflow}(j, s_t)$, где $r_{ij} = \cos(p_i, e_j)$ — релевантность, а штраф включается только при превышении 85% заполненности зала.
3. Вероятности выбора формируются как softmax с температурой τ, к рекомендациям добавляется альтернатива «отказ» с базовой вероятностью $p_{\text{skip}}$.
4. Случайно выбранный исход (либо id доклада, либо отказ) фиксируется; при выборе доклада соответствующий счётчик заполненности зала инкрементируется.

После прогона симулятор возвращает:
- последовательность шагов (`StepRecord`): id пользователя, id слота, выданные рекомендации, фактический выбор, релевантность выбранного, заполненность зала на момент выбора;
- агрегированную загрузку залов по (slot, hall).

### 3.3.2. Действующие параметры

| Параметр | Дефолт | Назначение |
|---|---|---|
| K | 2 | размер выдачи (top-K из 3 параллельных) |
| τ | 0.3 | температура softmax выбора |
| λ | 2.0 | штраф за переполнение в полезности |
| p_skip | 0.05 | базовая вероятность отказа |

Устойчивость к выбору параметров проверяется ablation-исследованием в подразделе 4.4.

### 3.3.3. Implementation notes

Симулятор работает в режиме **онлайн sequential greedy**: пользователи обрабатываются последовательно, и каждый следующий пользователь видит обновлённый load. Это согласовано с реальным режимом работы рекомендательного сервиса конференции (запросы приходят в неупорядоченные моменты времени), но отличается от полностью одновременной коллективной выдачи. Соответствующее уточнение внесено в постановку (см. подраздел 2.1).

## 3.4. Реализация рекомендательных политик

Все политики имеют единый интерфейс `__call__(user, slot, conf, state) -> List[talk_id]`. Это даёт возможность подключить любую политику в симулятор без изменения его кода.

### 3.4.1. Random
`src/policies/random_policy.py` — равномерная случайная выборка K докладов из доступных. Используется как baseline.

### 3.4.2. Cosine
`src/policies/cosine_policy.py` — top-K по $\cos(p_i, e_j)$ без учёта вместимости. Реализация представляет максимально упрощённый вариант семантической рекомендательной системы.

### 3.4.3. MMR (Maximal Marginal Relevance)
`src/policies/mmr_policy.py` — жадный отбор top-K с балансом релевантности и тематического разнообразия:

$$
\mathrm{MMR}(j \mid S) = \beta \cdot r_{ij} - (1 - \beta) \cdot \max_{l \in S} \cos(e_j, e_l), \quad \beta = 0.7
$$

### 3.4.4. Capacity-aware (rule-based)
`src/policies/capacity_aware_policy.py` — top-K по утилитарности, учитывающей загрузку зала:

$$
\mathrm{score}(j) = r_{ij} - \alpha \cdot \mathrm{load\_frac}(j, s_t), \quad \alpha = 0.5
$$

Жёсткая фильтрация: ресурсы с $\mathrm{load\_frac} \geq 0.95$ исключаются (если остаётся хоть один кандидат). Это реализует action masking в политике (а не в симуляторе) — соответствует структурному условию из подраздела 2.1.

### 3.4.5. Capacity-aware MMR
`src/policies/capacity_aware_mmr_policy.py` — гибрид MMR и capacity-штрафа:

$$
\mathrm{score}(j \mid S) = \beta \cdot r_{ij} - (1 - \beta) \cdot \max_{l \in S} \cos(e_j, e_l) - \alpha \cdot \mathrm{load\_frac}(j, s_t)
$$

с $\beta = 0.6, \alpha = 0.4$.

### 3.4.6. LLM-ranker без state-awareness
`src/policies/llm_ranker_policy.py` — ранжирование через LLM (по умолчанию `openai/gpt-4o-mini` через OpenRouter).

Промпт (упрощённо):
```
Ты ранжируешь доклады IT-конференции под пользователя.
Профиль: <текст>
Кандидаты: <id, title, category, abstract>
Верни JSON-массив id в порядке убывания приоритета.
```

Кэширование: ключ — SHA1-хэш `(base_user_id, slot_id, sorted(candidate_ids))`. Это даёт детерминированность вызова на одинаковых входах между сидами и репликами.

### 3.4.7. LLM-ranker с state-awareness
`src/policies/llm_ranker_state_aware_policy.py` — то же, что 3.4.6, но к каждому кандидату добавляется метка `low / medium / high / overflow` для соответствующего зала, и в системном промпте даётся явная инструкция учитывать загрузку:

```
Если зал отмечен overflow — туда направлять плохо: пользователь не попадёт.
Если зал отмечен high (>80%) — лучше выбрать менее загруженный близкий по релевантности.
Если зал отмечен low (<50%) — отличный кандидат при сопоставимой релевантности.
```

Кэш-ключ дополнительно включает signature загрузок (`tid:bucket` пары), что обеспечивает воспроизводимость на одинаковых состояниях.

## 3.5. Метрики оценки

`src/metrics.py` реализует метрики из подраздела 2.5:

- `overflow_rate(choice_only=True)` — основная метрика, доля переполненных (slot, hall) пар на 12 слотах с реальным выбором между параллельными докладами.
- `overflow_rate(choice_only=False)` — для контекста, включая 4 single-talk слота, в которых переполнение зависит только от размера аудитории.
- `hall_utilization_variance` — дисперсия загрузки залов внутри слота, усреднённая по 12 слотам с ≥2 параллельными залами.
- `mean_user_utility` — средняя релевантность фактически выбранных докладов (для отказов 0).
- `hall_load_gini` — коэффициент Джини суммарной загрузки залов за всю конференцию.

Дополнительно: `skip_rate`, `mean_overload_excess`.

## 3.6. Конфигурация экспериментальных прогонов

`scripts/run_experiments.py` принимает параметры через CLI:
- `--personas` — имя файла набора пользователей (`personas_x3` для 900);
- `--seeds` — список сидов;
- `--K`, `--tau`, `--lambda-overflow`, `--p-skip` — параметры симуляции;
- `--with-llm`, `--with-llm-sa` — включение LLM-rankers;
- `--llm-model`, `--llm-budget`, `--llm-budget-sa` — параметры LLM.

Результаты сохраняются в `results/results.json` (детальные числа по каждому прогону) и `results/summary.md` (агрегированная таблица mean ± std).

## 3.7. Прекэширование LLM-ranker (без state)

`scripts/precompute_llm_rankings.py` — асинхронный сценарий, который для всех уникальных пар (user_id_base, slot_id) делает запрос к LLM-ranker'у и сохраняет результат в кэш `logs/llm_ranker_cache.json`. Использует `asyncio.Semaphore(30)` для ограничения параллелизма, что позволяет 4800 уникальных вызовов выполнить за 3.5 минуты вместо ~50 минут последовательно.

State-aware LLM-ranker не прекэшируется заранее, поскольку состояние зависит от хода симуляции; вместо этого состояние бакетируется на 4 уровня (`low / medium / high / overflow`), что обеспечивает высокий cache hit rate в пределах одного и нескольких прогонов.

## 3.8. Конфигурация ablation studies

`scripts/ablation.py` — прогон сетки параметров. Для каждого из трёх sweep-ов (τ, λ, K) проводится отдельный полный цикл из 6 политик × 3 сидов на полной популяции. Результаты сохраняются в три отдельных JSON-файла и объединённую таблицу `results/ablation_summary.md`.
