## 📌 Контекст и изменения в требованиях

В последних версиях критериев защиты произошли заметные изменения:

- **Слепое рецензирование** — рецензент больше не выбирается напрямую научником, что добавляет объективности.
- **Валидация стала ключевым критерием** — результаты должны быть не просто продемонстрированы, а доказаны и воспроизводимы.
- **Импакт и новизна** теперь можно закрывать не только числами, но и качественной аблацией.

---

## 🧩 Три типа задач валидации в ВКР

### 1. Научная валидация

Классический академический подход:

- Формулируем нулевую гипотезу
- Выбираем целевую метрику
- Фиксируем критерий приёмки
- Считаем p-value, бутстрап, доверительные интервалы

### 2. Инженерные метрики

Даже если нулевая гипотеза не подтвердилась — сервис может работать отлично. Важно показать:

- Human-in-the-loop валидацию
- Производительность на размеченных данных
- Финансовые показатели (стоимость, ROI)

### 3. Новизна и импакт

Самая сложная часть. Закрывается через **качественную аблацию** — исследование, которое доказывает, что именно ваш вклад влияет на результат. Работы, где аблации нет, автоматически теряют в оценке.

## 🔬 Дизайн экспериментов по Фишеру

Хороший эксперимент включает **шесть компонентов**:

| Компонент | Описание |
| --- | --- |
| **Research question** | Гипотеза, которую хотим проверить |
| **Данные** | Датасет, который масштабируется на реальный мир |
| **Метрика** | Дистинктивная мера качества работы системы |
| **Focal property** | Ось оценки — свойство системы, которое оцениваем |
| **Дизайн** | Среда, железо, seed, количество экспериментов |
| **Decision criteria** | Критерий приёмки — при каких значениях эксперимент успешен |

> 💡 **Ключевой принцип:** формулируйте ожидаемый результат *до* запуска эксперимента. Иначе возникает **p-hacking** — подгонка метрики под полученные результаты.
>

## 📊 Структура датасета

Хорошая тест-система состоит из трёх типов данных:

```
📂 Датасет
├── 🥇 Golden set       — идеальные экспертные примеры
├── ⚠️  Adversarial set  — примеры, где система ломалась
└── 🔁 Regression set   — кейсы из прошлых ошибок
```

**Требования к датасету:**

- Чистота — нет утечек (leaks), нет семантических дублей
- Достаточность — примеров хватает для экстраполяции на реальный мир
- Разнообразие — данные покрывают распределение реальных запросов

---

## 📐 Робастность (Robustness)

Система должна не просто хорошо работать на целевом домене, но и **не деградировать** на остальных.

**Что проверяем:**

- Поведение на нецелевых доменах
- Устойчивость к шуму и prompt injection
- Безопасность (особенно для агентных систем)

**Как проверяем:**

- Оффлайн: golden set из других доменов
- Онлайн: тематическая кластеризация логов + дрифт-анализ
- Алертинг: LLM-тег на каждое сообщение (целевое / нецелевое / атака)

## 📈 Фреймворки оценки

### HELM (Stanford)

Оценивает систему по **нескольким осям** одновременно — не только качество, но и латентность, стоимость, безопасность.

**Зачем это нужно для ВКР:** показывает честную картину trade-offs. Например:

> *«Модель работает в 2 раза быстрее, теряя 0.1% в точности — берём её»*
> 

### HuggingFace Evaluation Guidebook

Практические тезисы по построению показательных датасетов и оценке моделей. Рекомендуется к прочтению.

## 🤖 Использование LLM как судьи (LLM-as-a-Judge)

Обычные модели **не подходят** для объективной оценки — они склонны хвалить пользователя из-за RLHF-обучения.

**Что работает:**

- **LLM Judge через фреймворки** (напр. JUDEVAL, UpTrain) — с чёткими критериями, дискретной шкалой, few-shot примерами
- **Кросс-оценка** несколькими моделями (каждая смотрит на разные аспекты)
- **Калибровка** на человеческой разметке — показываем модели примеры правильных и неправильных ответов
- **Chain-of-Thought обязателен** — просим модель объяснять оценку

**Настройки Judge-промпта:**

```
- Роль: «Ты рецензент топ-конференции / научный руководитель»
- Задача: найти N причин, почему это не работает
- Температура: ≈ 0 (детерминированность)
- Шкала: дискретная с примерами
- Shuffle датасета перед каждым прогоном
```

## ❌ Типичные ошибки валидации

| Ошибка | Почему это плохо |
| --- | --- |
| Слишком много метрик | Бессмысленно, непонятно на что опираться |
| Нет доменной экспертизы | Шум в данных искажает оценку качества |
| Метрика без связи с импактом | +2% точности ценой года работы — зачем? |
| Некалиброванные LLM-метрики | Модели прыгают по шкале без заземления |
| Нет разбора ошибок | Рецензент спросит — и вы не знаете ответа |
| Краудсорсная разметка без контроля | Вопросы к качеству аннотации |

## 🔍 Разбор ошибок — ваша сильная позиция

> *«Где ваша система ломается?»* — один из главных вопросов на защите.
> 

Ответ **«не входило в спектр моих задач»** — слабая позиция.

**Как правильно:**

1. Собрать худшие примеры из датасета регрессий
2. Разметить и кластеризовать ошибки
3. Показать: *почему ломается, что делают конкуренты, каков future work*

Знание своих слабых мест = контроль над нарративом защиты.

## 💰 Про бизнес-метрики и инженерные решения

Если ваш проект — продуктовый, ключевая метрика — **деньги**.

**Что показывать:**

- Корреляция технических метрик с бизнес-результатом
- Период окупаемости решения
- Стоимость поддержки vs. качество
- Сравнение с open-source альтернативами (на обезличенных данных)

> 💡 Простое решение, которое понятно зарабатывает деньги, часто лучше сложного с чуть лучшими метриками.
>

## 📋 Вопросы, которые задаст рецензент

Подготовьте ответы заранее:

- [ ]  Почему именно эта метрика?
- [ ]  С чем сравниваете? Почему это baseline?
- [ ]  Как собирали тест? Есть ли утечки и дубли?
- [ ]  Статистическая значимость — сколько сидов, какой тест?
- [ ]  Как валидировали разметку? Какой inter-annotator agreement?
- [ ]  Где ваша система ломается? Покажите 5 худших примеров.
- [ ]  Почему этот график? Почему не показываете другие модели?
