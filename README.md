# Магистерская ВКР: сценарный полигон для оценки риска перегрузки залов конференции

Магистерская работа в ИТМО, программа Industrial AI, защита 2026.

**Тема:** «Разработка интеллектуальной системы поддержки формирования программы конференции с учётом ограничений вместимости параллельных сессий и согласованности рекомендаций между участниками».

**Заказчик:** компания JUG, конференции Mobius (мобильная разработка) и Heisenbug (тестирование).

> ⚠ Главный документ проекта — [`PROJECT_STATUS.md`](PROJECT_STATUS.md). Любая новая сессия Claude Code обязана прочитать его первым. Этот README — навигация по репо.

## Идея в двух абзацах

Реальных данных посещаемости конференций нет — это проверенный факт литературы. Поэтому работа не предсказывает реальную посещаемость и не утверждает, что какой-то рекомендатель «лучший». Вместо этого разработан **сценарный аналитический полигон**: организатор конференции до её начала задаёт сценарии вместимости залов, уровня доверия аудитории к рекомендации (compliance) и политику рекомендаций — и получает таблицу относительных рисков перегрузки залов и устойчивости политик.

Аналогия: не прогноз погоды, а стресс-тест здания. Мы не утверждаем, какой именно ветер будет, но проверяем, какие конструкции уязвимы при разных ветрах.

## Главный измеренный результат

E2 falsification на Mobius LLM-симе (4 политики, n=50, K=2, seed=42):

| Политика | OF_choice | mean_overload_excess | hall_var_mean | mean_user_utility |
|---|---|---|---|---|
| no_policy | 0.502 | 0.211 | 0.368 | 0.8264 |
| cosine | 0.496 | 0.232 | 0.359 | 0.8285 |
| cosine_capacity_filter | 0.507 | 0.157 | 0.218 | 0.8276 |
| cap_aware_mmr | 0.453 | 0.150 | 0.223 | 0.8272 |

Артефакт: `experiments/results/llm_agents_mobius_2025_autumn_n50_falsification_4pol.json`.

**Содержательное наблюдение (illustrative, single-seed):** в этом конкретном прогоне `cosine_capacity_filter` и `cap_aware_mmr` практически совпадают по `mean_overload_excess` и `hall_var_mean`; разрыв `cap_aware_mmr − cosine_capacity_filter` по `OF_choice` составляет 5 п.п. на одном seed. Полезность участников практически идентична у всех четырёх политик. Основной эффект на метрики перегрузки даёт операциональная маска занятости, а не алгоритмическое усложнение MMR.

Этот результат — отправная точка нового нарратива работы (см. `PROJECT_STATUS.md` §6). Ключевое сравнение `policy × capacity × compliance` будет получено в preflight scenario matrix (см. ниже).

## Что НЕ защищается в работе

См. `PROJECT_STATUS.md` §1 и стоп-лист в memory. Кратко:

- «лучший персональный рекомендатель докладов»;
- «cap-aware MMR — победитель»;
- «предсказание реальной посещаемости»;
- B1 / accuracy@1 = 0.918 как валидация preference-модели (есть утечка через memberships);
- cross-domain Spearman ρ как «валидация против реальности» (это самосогласованность, не валидация);
- CoSPLib как attendance / preference / capacity benchmark;
- UMass как конференцию.

## Принятые методологические решения

- **Эмбеддер:** `intfloat/multilingual-e5-small` с query/passage префиксами.
- **Group split по `persona_id`** в обучении preference-модели.
- **Метрика `mean_overload_excess`:** только слоты с ≥ 2 залами (исключён keynote-вклад).
- **Capacity на Mobius / Demo Day:** задана исследователем как параметр сценария, не реальные данные конференции.
- **Калиброванный compliance:** трёхтипная модель из Meetup-RSVP — `compliant 71.7% / star-chaser 21.3% / curious 7.0%`.
- **Single-async симулятор:** `simulate_async(slot_concurrency)` — параллельные слоты для LLM-политик, последовательные для остальных.
