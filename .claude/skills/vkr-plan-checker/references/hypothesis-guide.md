# Гайд по формулировке гипотез

## Два типа гипотез

---

### Тип 1 — Метрическая гипотеза

```
[Предлагаемый метод/подход] превзойдёт [baseline]
по [метрика] на [датасет/условие]
не менее чем на [порог]
```

Четыре обязательных элемента — если хотя бы одного нет, гипотеза неполная.

**1. Что сравнивается**
Твой метод vs. что конкретно? Baseline должен быть честным — лучший существующий метод или наиболее очевидное решение «в лоб».

Плохо: «по сравнению с существующими методами»
Хорошо: «по сравнению с GPT-4 zero-shot» / «по сравнению с nnU-Net» / «по сравнению с наивным парным сравнением»

**2. Метрика**
Как измеряется «лучше»? Метрика должна соответствовать задаче и этапу.

| Тип задачи | Примеры метрик |
|---|---|
| Классификация | F1, precision/recall, AUC-ROC |
| Генерация текста | ROUGE, METEOR, BERTScore, human eval |
| Детекция/сегментация | Dice, IoU, mAP |
| Ранжирование | NDCG, MRR, precision@k |
| Продуктовые задачи | Retention, NPS, CSAT, conversion |
| Образовательные | Accuracy gain, engagement rate, NPS |
| Системные | Latency (p95), throughput, recall@k |

**3. Условие**
На каких данных, в каком контексте, при каких ограничениях?

Плохо: «на тестовых данных»
Хорошо: «на публичном датасете IU X-Ray» / «при ≤ 50 размеченных примерах» / «при нагрузке 2000 объявлений в минуту»

**4. Порог**
Насколько лучше, чтобы это было значимо? Порог нужно обосновать.

Плохо: «улучшит результат»
Хорошо: «не менее чем на 8 п.п. — что соответствует клинически значимому улучшению, поскольку разрыв между текущим SOTA и уровнем эксперта составляет ~15 п.п.»

### Тип 2 — Архитектурная / дизайновая гипотеза

Используется когда новизна — в архитектуре, дизайне данных или подходе к задаче, а не только в сравнении метрик.

```
Подход [X] решает проблему [Y],
которую [существующий подход Z] не решает,
потому что [конкретное ограничение Z].
Это подтверждается через [как проверяем].
```

Три обязательных элемента: предлагаемый подход, проблема которую он решает, конкретное ограничение существующего.

**Хороший пример — индустриальный трек:**
> «Трёхэтапная система дедубликации (генерация кандидатов → ранжирование → кластеризация) обеспечит точность F1 > 0.85 при обработке 1M+ объявлений в день — в отличие от наивного парного сравнения, которое неприменимо при такой нагрузке из-за квадратичной вычислительной сложности.»

**Хороший пример — научный трек:**
> «Специализированный доменный корпус (>50K пар текст-реферат) позволит дообучить трансформерные модели и существенно повысить ROUGE-L по сравнению с zero-shot на общих корпусах — поскольку существующие датасеты преимущественно англоязычны и не содержат отраслевой терминологии.»

**Хороший пример — образовательный трек:**
> «AI-ассистент с Socratic-подсказками снизит долю повторных ошибок того же типа с 67% до < 40% — по сравнению с контрольной группой без ассистента — поскольку существующие инструменты (Copilot, ChatGPT) дают готовый ответ и не развивают навык отладки.»

## Примеры метрических гипотез по трекам

### Индустриальный
> «RAG-система с кэшированием эмбеддингов обеспечит latency < 1.5 сек (p95) при precision@3 > 0.85 на корпоративном датасете из 50 000 документов — по сравнению с baseline Elastic Search + GPT-4 zero-shot (latency ~5 сек, precision@3 ~0.79).»

> «Мультиагентная RAG-система с ReAct-паттерном обеспечит корректность ответов > 90% при latency < 3 сек на доменном корпусе — подтверждается task-based оценкой на 20 сценариях и пользовательской оценкой (CSAT > 90%, NPS > +50) на выборке > 20 пользователей. Baseline: та же LLM без RAG (40% корректных ответов).»

> «RAG-система на базе ModernBERT (контекст 8192 токена) обеспечит F1 > 0.8 на доменном корпусе — по сравнению со стандартным BERT-ретривером (контекст 512 токенов). Оценка раздельно: retrieval (precision@k) и generation (faithfulness, correctness через DeepEval).»

### Стартап
> «Независимые фотографы готовы платить 2 990 руб/мес за AI-инструмент постобработки, если он сокращает время с 4–6 часов до < 40 минут. Подтверждается при retention > 60% через 3 месяца и NPS > 40 на выборке из 30 пользователей.»

### Научный
> «Модель с механизмом cross-modal attention достигнет BLEU-4 > 0.45 (vs. 0.38 у SOTA) и снизит долю клинических ошибок с 23% до < 15% на датасете IU X-Ray — по независимой оценке двух радиологов-экспертов.»

> «Slot-fusion модель с мультимодальными описаниями (текст, аудио, видео, Action Units) улучшит weighted-F1 на двух независимых диалоговых датасетах по сравнению с текстовым baseline. Ablation study: вклад каждой модальности оценивается отдельно.»

> «Мультимодальная LLM, дообученная на 13-канальных мультиспектральных данных дистанционного зондирования, превзойдёт RGB-модели по zero-shot классификации на стандартном бенчмарке (Accuracy: 0.64 vs. 0.58) — потому что стандартные VLM обучены только на RGB и не воспринимают мультиспектральную информацию.»

### AI в образовании
> «AI-ассистент для дебаггинга с Socratic-подсказками снизит долю повторных ошибок того же типа с 67% до < 40% у первокурсников ИТМО — по сравнению с контрольной группой без ассистента (N = 60, 2 группы по 30).»

## Частые ошибки

| Слабая формулировка | Проблема | Исправление |
|---|---|---|
| «Система будет работать эффективно» | Нет ни одного из 4 элементов | Переформулировать полностью |
| «Метод улучшит результаты» | Нет метрики, условия, порога, baseline | Добавить все три |
| «Точность > 90%» | Нет baseline — непонятно, хорошо это или нет | «...превзойдёт baseline X (83%) на датасете Y» |
| «Предложен перспективный подход» | Не гипотеза вообще | Заменить на проверяемое утверждение |
| «Использование нового датасета улучшит качество» | Тип 2, но без указания почему существующие не работают | Добавить конкретное ограничение существующих датасетов |

### Anomaly detection / Active learning (научный или индустриальный)
> «Semi-supervised метод на основе contrastive learning + active learning достигнет Detection TPR > 0.68 и ACC > 0.73 при разметке < 5% данных — в отличие от unsupervised (PCA+Kmeans, TPR < 0.36) и supervised (требует 100% разметки). Валидация на Tennessee Eastman benchmark.»

### Мультиагентная система / AI-агент (индустриальный или научный)
> «Мультиагентный pipeline (STT → LLM → TTS → анимация) обеспечит автономный стриминг с временем реакции < 3 сек и MOS голоса > 4.0 — в отличие от существующих решений, требующих постоянного участия человека. Ablation: замена каждого компонента (3 альтернативные TTS-системы разного качества и latency) влияет на MOS и end-to-end latency.»

### CV + Edge-деплой (индустриальный)
> «Лёгкая модель (MobileNetV3) с INT8-квантизацией на NPU edge-устройства обеспечит f1-score > 0.93 при > 140 FPS и стоимости устройства < 10 000 руб — в отличие от серверных решений, требующих постоянного канала связи (latency > 5 сек, зависимость от сети).»

> «Метод распознавания радужки на основе ключевых точек (KP-RPE) обеспечит TAR > 95% при FAR = 0.01% с вычислительной сложностью < 1 GFLOP — в отличие от полносвёрточных моделей (ArcFace, AdaFace), требующих > 5 GFLOP. Валидация на бенчмарке IREX.»

### Создание датасета / бенчмарка (научный или индустриальный)
> «Специализированный мультиязычный корпус (> 50K пар текст-реферат) для торговых новостей позволит fine-tuning трансформерных моделей и существенно улучшит ROUGE-L по сравнению с zero-shot на общих корпусах — потому что существующие датасеты не содержат торговой лексики и русскоязычных текстов.»

> «End-to-end модель Swin Transformer + GPT-2, обученная на > 100K пар (PDF → Markdown) из arXiv, превзойдёт лучший существующий open-source baseline по BLEU (> 0.60 vs 0.43) и WER (< 0.45 vs 0.72) — благодаря более качественному pipeline подготовки данных и расширенному контексту декодера (4096 vs 1024 токенов).»

### Ранжирование / Information Retrieval (научный или индустриальный)
> «Domain-specific языковая модель, предобученная на данных платформы (27B токенов), превзойдёт general-purpose модели сопоставимого размера по NDCG@10 на задаче ранжирования ответов — потому что доменный токенизатор лучше обрабатывает техническую терминологию и code-mixed тексты.»

### Инструмент / платформа с user study (индустриальный или образовательный)
> «Двумерный пространственный интерфейс для ноутбуков ускорит выполнение задач data science на > 15% и значимо улучшит восприятие AI-ассистента (Wilcoxon p < 0.01) — по сравнению с линейным Jupyter. User study: N > 15, within-subjects, Scheirer-Ray-Hare test.»

> «Интерактивный симулятор A/B-тестирования с LLM-подсказками снизит время освоения на > 25% по сравнению с лекционным форматом — подтверждается pre/post тестом на выборке > 20 студентов и SUS > 70.»

## Для научного трека: H₀ и H₁

**H₀ (нулевая):** предлагаемый метод не отличается от baseline по метрике M на датасете D.
**H₁ (альтернативная):** предлагаемый метод превосходит baseline на величину > threshold.

Это делает эксперимент воспроизводимым и проверяемым.
