# Чеклист плана и постановки задачи — 12 критериев + бонус

Для каждого: ✅ выполнен / ⚠️ частично / ❌ отсутствует.

**Структура:** критерий 0 (допуск) и критерий 12 (раскрытие AI-инструментов) — обязательные «ворота», без них работа не допускается к защите. Критерии 1–9 — основа любой ВКР. Критерий 10 (System Design) применяется для Industrial / production-работ. Критерий 11 (воспроизводимость) — сквозной. Бонус (внешняя валидация) не входит в счёт, но сильно усиливает работу.

## 0. Допуск: соответствие программе и теме

✅ AI/ML является центральным вкладом работы, а не инструментом «для галочки». Тема заявлена и раскрыта — все главы работают на доказательство именно этой темы.
❌ AI упомянут в одном разделе из трёх. Или: тема «Система X», а три главы — про MLOps-инфраструктуру.

Правило: убери AI-часть из работы. Остаётся что-то самостоятельно ценное для программы «Искусственный интеллект»? Если нет — AI-часть нужно усилить или перепозиционировать.

## 1. Проблема сформулирована с реальным ущербом и без технологий

✅ «Врачи тратят 3–4 часа на ручную разметку снимков. При высокой нагрузке вероятность ошибки вырастает до 15%.»
❌ «Существующие методы компьютерного зрения имеют ограниченную точность.»

Правило: убери все названия методов и технологий из описания проблемы. Осталось что-то осмысленное с измеримым ущербом? Значит, проблема сформулирована правильно.

---

## 2. Объяснено, почему существующие решения не работают

✅ «Метод X достигает точности Y, но на данных из области Z точность падает до W — авторы сами указывают на это ограничение.»
❌ «Существующие методы недостаточно точны.»

Правило: должна быть конкретная цифра, ограничение или условие, при котором существующее решение ломается применительно к твоей задаче. Это же обосновывает новизну.

---

## 3. Постановка задачи на двух уровнях

✅ «Бизнес-задача: сократить время разметки с 12 до 2 минут. Техническая задача: few-shot сегментация медицинских изображений.»
❌ «Цель — разработать систему автоматической сегментации.»

Правило (требование ИТМО): постановка на языке бизнес-требований + в терминах технической задачи (ML/DA).

---

## 4. Гипотеза проверяема и содержит нужные элементы

**Тип 1 — метрическая:** что сравнивается + метрика + условие + порог.

✅ «Fine-tuned U-Net с domain adaptation превзойдёт nnU-Net zero-shot по Dice Score на датасете X не менее чем на 8 п.п. при ≤ 50 размеченных примерах.»
❌ «Предполагается, что система будет работать эффективнее существующих.»

**Тип 2 — архитектурная/дизайновая:** подход + проблема которую он решает + почему существующий не решает.

✅ «Трёхэтапный пайплайн (кандидаты → ранжирование → кластеризация) обеспечит дедубликацию 1M+ объявлений в день — в отличие от наивного парного сравнения, непримениого при такой нагрузке из-за квадратичной сложности.»
❌ «Предлагается использовать новую архитектуру системы.»

---

## 5. Указан критерий успеха

✅ «Работа считается успешной, если Dice Score > 0.85 на hold-out выборке при ≤ 50 примерах.»
❌ Нет явного порога — непонятно, когда гипотеза подтверждена, а когда нет.

## 6. Валидация многоуровневая, метрики обоснованы

✅ Есть несколько независимых слоёв оценки: алгоритмический (метрики на hold-out) + системный (latency/throughput, **2–3 уровня нагрузки stepwise**) + пользовательский (до/после / обратная связь / LLM-as-a-Judge). Выбор каждой метрики обоснован и соответствует типу задачи. При дисбалансе классов — **per-class F1** обязательно (не только средний weighted F1).
❌ Только один слой оценки. Или: метрика не обоснована, выбрана «по умолчанию». Или: средняя метрика «по больнице», скрывающая провалы на редких классах.

Правило 2026: «я реализовал X» — это половина работы. «Я реализовал X, доказал что он решает проблему Y лучше Z по метрике M, и вот независимая оценка» — полная работа.

**Сильные паттерны валидации:**
- **Композитная метрика с обоснованием весов** (через grid search или экспертную разметку), а не «возьмём метрики и усредним».
- **Эмпирическая критика выбора варианта метрики** — например, NDCG с linear gain «saturates» → переход на exponential gain с иллюстрацией.
- **LLM-as-a-judge с двумя независимыми судьями** (разные модели от разных провайдеров) и сверка их разногласий — single-judge выдаёт радикально разные оценки разных моделей.
- **Bootstrap (10 000+ итераций) + permutation test + Bonferroni** для статистической оценки различий конфигураций.
- **κ Флейса / κ Коэна для inter-rater agreement** при экспертной разметке + протокол разрешения конфликтов (мажоритарное голосование при <3 совпадениях из 5 экспертов).
- **Cross-dataset / zero-shot валидация** (модель обучена на A → тестируется на B и наоборот) — сильнее, чем train/val split на одном датасете.
- **Анти-leak проверка датасета** — доля дословных предложений из text в summary / extractive leak в абстрактивных задачах.
- **Statистический анализ ошибок по подгруппам** (Mann-Whitney U / Wilcoxon) с фильтром min sample size — иначе выводы по подгруппам с n<10 ненадёжны.

**Для RAG-систем — валидация на двух уровнях:**
- **Retrieval-качество** (отдельно от generation): precision@k, recall@k, F1 на retrieval — оценивается ли ретривер независимо от LLM?
- **Generation-качество:** корректность ответа, галлюцинации, faithfulness — используются ли специализированные метрики (RAGAS, DeepEval, LettuceDetect)?
- **Сквозная оценка:** task-based accuracy на реальных сценариях.
- **Онлайн-мониторинг:** трейсинг (Langfuse/LangSmith), мониторинг drift, отслеживание стоимости API-вызовов.

Антипаттерн: оценивать RAG только по итоговому ответу LLM, не проверяя качество retrieval отдельно. Если ретривер возвращает нерелевантные чанки, даже лучшая LLM не спасёт — но ошибку будет сложно диагностировать.

**Для мультимодальных работ — ablation по модальностям:**
- Каждая модальность (текст, аудио, видео, мета-данные и т.д.) оценена отдельно — какой вклад она вносит?
- Fusion-стратегия обоснована (early / late / slot-fusion / cross-attention) — сравнивались ли альтернативы?
- Если создан собственный датасет — описан процесс сбора, очистки, дедупликации, аугментации, разбиения на train/test?
- Если используется fine-tuning foundation model — обоснован выбор модели, описаны гиперпараметры (LoRA rank, learning rate, batch size, epochs)?

Антипаттерн: добавить модальность и показать улучшение итоговой метрики — но не показать, что именно эта модальность внесла вклад. Без ablation — непонятно, нужна ли она или результат случаен.

**Для работ с созданием датасета / бенчмарка:**
- Проанализированы ли существующие датасеты — и каждый получил вердикт «почему не подходит»?
- Описан ли pipeline сбора и подготовки данных (источники, очистка, валидация, формат)?
- Опубликован ли датасет для воспроизводимости (HuggingFace, Zenodo, GitHub)?
- Используются ли несколько метрик оценки (не одна), включая human eval или LLM-as-a-Judge?

**Для работ с ранжированием / information retrieval:**
- Разделены ли задачи retrieval и ranking?
- Сравниваются ли domain-specific и general-purpose модели?
- Есть ли error analysis — на каких категориях/тегах модель ошибается?
- Обоснован ли выбор архитектуры (bi-encoder vs cross-encoder vs ColBERT)?

**Для CV-работ с оптимизацией и edge-деплоем:**
- Есть ли сравнение нескольких архитектур моделей (а не только одна)?
- Описан ли pipeline оптимизации: pruning → quantization → export (ONNX/TensorRT/RKNN)?
- Приведена ли таблица: точность vs. скорость vs. размер модели при разных уровнях оптимизации?
- Обоснован ли выбор edge-устройства (сравнение альтернатив по цене, производительности, энергопотреблению)?
- Есть ли экономическая оценка: сколько стоит решение, какой ROI, при каком масштабе окупается?

Антипаттерн: задеплоить модель на edge-устройство и показать что «работает» — но не показать trade-off между качеством и скоростью, не обосновать выбор устройства, не оценить экономику.

**Для работ с инструментами / платформами / user study:**
- Проблема подтверждена данными (опрос, статистика, бенчмарк), а не только интуицией автора?
- User study: описан дизайн (within/between subjects), размер выборки, контроль порядка?
- Статистические тесты корректны для размера выборки (непараметрические для N < 30)?
- Приведены и количественные (время, ошибки), и качественные (Likert-шкалы, интервью) результаты?
- Бенчмарк воспроизводим (Docker, зафиксированные промпты, seeds)?

Антипаттерн: провести user study с 5 участниками без контрольной группы и заявить «значимое улучшение». Или: создать бенчмарк, но не описать процесс сбора данных и не обеспечить воспроизводимость.

**Для работ с anomaly detection / active learning:**
- Есть ли сравнение трёх режимов: unsupervised, semi-supervised, supervised?
- Показан ли trade-off: объём разметки vs. качество (при 1%, 5%, 10%, 100% разметки)?
- Описана ли стратегия active learning — как выбираются примеры для разметки?
- Используется ли стандартный benchmark (Tennessee Eastman, SWAT, WADI)?

**Для мультиагентных систем / AI-агентов:**
- Описана ли архитектура как pipeline с обоснованием выбора каждого компонента?
- Есть ли ablation: замена каждого компонента (STT, LLM, TTS) → как меняется результат?
- Есть ли количественные метрики (latency, uptime, MOS, retention), а не только «работает»?
- Валидация не только на соревновании — есть ли A/B-тесты, пользовательские метрики?
- Для voice-агентов: измерена ли **end-to-end latency от звука до звука**, а не только модельная? Что с long-running stability (>1 час непрерывной работы)?
- Для multi-agent с composite scoring (например, формула токсичности с весами): обоснованы ли веса (grid search, экспертная разметка, или хотя бы анализ чувствительности к границам)?

**Для Document AI / OCR / vision-to-markup задач:**
- Указано разрешение входа модели и обоснование выбора?
- Описан размер контекста декодера и его ограничения (для seq2seq)?
- Pipeline создания ground truth прозрачен (конкретные инструменты конверсии — например, .tex → промежуточный формат → итог) с указанием рисков каждого шага?
- Сравнение с конкретным OS-аналогом, а не «без OCR» как baseline?
- Учтён trade-off seq2seq encoder-decoder vs multimodal LLM (с явным обсуждением «качество vs стоимость инференса / закрытость»)?
- Метрики OCR-специфичные: BLEU + CER + WER (не одна), плюс PPS как метрика производительности?

**Для recommender systems / matching / entity resolution:**
- Семейство метрик с обоснованием k: HitRate@k, Precision@k, Recall@k, mAP@k, NDCG@k?
- **Coverage** измерена (доля каталога / уникальных объектов в выдаче) — иначе модель может выдавать одни и те же топ-айтемы?
- Two-tower vs cross-encoder trade-off обсуждён (precompute embeddings vs accuracy)?
- Двухступенчатая архитектура retrieval + reranking — оценка каждого этапа отдельно?
- Сравнение бейзлайнов: content-based / collaborative filtering / hybrid?
- Hard negative mining (через TF-IDF / Bayesian / similarity threshold) обоснован — или объяснено, почему случайных негативов достаточно?
- Для duplicate detection / entity resolution: метрики на двух уровнях — pair-wise (P/R/F1) И cluster-wise (ARI, Fowlkes-Mallows)?
- Описан процесс получения gold-разметки (несколько итераций, шкала «ловушек», протокол разрешения межаннотаторских конфликтов)?
- Экономика ошибки: false positive vs false negative — какая дороже и почему?

**Для anti-cheat / trust & safety / классификации с асимметричной стоимостью ошибок:**
- Первичная метрика — ROC-AUC или PR-AUC (не accuracy/F1, потому что классы сильно несбалансированы)?
- Зафиксирован ли operating point с явным порогом FP-rate (для bans-as-a-service это критично)?
- Решение объяснимо для апелляции (не «чёрный ящик»)?
- Тестирование устойчивости к adversarial behavior (например, sandbagging, evasion)?
- Этическая рамка: «инструмент для аналитика» vs «автоматический судья»?
- Если используется ансамбль ML-моделей — карточка каждой подмодели (версия, чекпоинт, собственная точность)?

**Для биометрических систем / face recognition:**
- Метрика TAR@FAR=0.01% (или иной operating point по FAR) — а не средняя accuracy?
- Open-set vs closed-set оценка различаются явно (система должна не только распознавать своих, но и корректно отвергать незнакомцев)?
- Указан ли GFLOP/compute-cost при сравнении эффективности (не только accuracy)?
- Использован ли отозванный датасет (например, MS-Celeb-1M) — указано ли это явно?
- Если задача — long-tail с миллионами классов: использован ли PartialFC или аналог?

**Для GraphRAG (расширение RAG-чеклиста):**
- Сравнение GraphRAG vs наивный RAG обосновано доменом (структурированный каталог, сложные связи)?
- Схема графа явно описана (узлы, рёбра, атрибуты, направленность, флаг embeddable)?
- Язык запросов выбран и обоснован (Cypher / SPARQL / GQL)?
- Механизм перевода NL → запрос (LLM генерирует Cypher) описан с примерами + валидация синтаксиса перед выполнением?
- Уровни сложности вопросов в golden dataset (1–4) с покрытием каждого?
- Онтология построена совместно с domain-experts (а не «из головы»)?

**Для action recognition / video classification (включая sign language):**
- Метрики: per-instance И per-class accuracy top-k (k=1,5,10) при дисбалансе классов?
- 3 стратегии слияния модальностей различены (раннее / промежуточное / позднее) и одна выбрана с обоснованием?
- Если используется псевдо-разметка (HRNet/OpenPose поверх RGB) — есть ли анализ ошибок генератора на доменных данных?
- Format requirements для входа явно описаны (полный обзор, освещение, фон, стабильность камеры)?

**Для multi-object tracking / спортивной аналитики:**
- HOTA = DetA × AssA измерена (не только mAP детекции)?
- Связка детектор + трекер (например, YOLO + ByteTrack) — сравнение режимов «только bbox geometric / +ReID / +frame_skip» проведено?
- frame_skip — предмет ablation с обоснованием через среднее перемещение объекта между кадрами?
- Бизнес-метрики поверх CV (FS, DF, события домена) — переход от технических метрик к продуктовым KPI?

## 8. Ограничения честно описаны

✅ Отдельный раздел с **минимум 3 конкретными ограничениями** по разным осям: данные / метод / условия применимости. Каждое ограничение явно ведёт к направлению future work по формуле «X не покрыто, потому что Y → future work: Z».
❌ Ограничения отсутствуют. Или: «работа имеет некоторые ограничения, которые будут устранены в будущем» — без конкретики. Или: упомянуто только одно ограничение в общих словах. Или: ограничения спрятаны в заключение одним предложением.

Шаблон сильного раздела ограничений — несколько категорий по разным осям:
- **Технические:** масштабируемость, производительность, ресурсные ограничения
- **Эмпирические:** размер выборки, длительность исследования, воспроизводимость
- **Доменные / по данным:** «обучено только на X — обобщающая способность на Y не исследована», непокрытые языки, модальности, форматы
- **Метода:** где система ломается, при каких сценариях деградирует, что не превзошло SOTA

Правило: сильные ВКР выносят ограничения в отдельный раздел. Честный анализ ограничений = признак зрелости, а не слабости. Каждое ограничение явно ведёт к направлению future work по формуле «X не покрыто, потому что Y → future work: Z».

## 10. System Design прослеживается от проблемы к архитектуре (Industrial / production-работы)

✅ В тексте есть явная цепочка: **проблема → функциональные и нефункциональные требования (latency, throughput, масштабируемость, отказоустойчивость) → архитектурные решения → trade-offs (что выбрали и почему отвергли альтернативы)**. Архитектура описана в виде диаграммы (C4, ERD, Data Flow), а не списка технологий.
❌ «Стек вместо архитектуры» — слайд / абзац с логотипами Docker, Kubernetes, Kafka без обоснования. Или: архитектура есть, но trade-offs не описаны — непонятно, почему именно так.

Правило: диаграмма с логотипами — не архитектура. Архитектура — это **цепочка решений с обоснованием**, начинающаяся от требований.

**Пятиуровневый System Design** (от слабого «стек логотипов» к сильному):
1. **Обоснование каждого компонента** — что он делает, какие альтернативы рассмотрены, почему выбран именно этот
2. **API-контракты** между компонентами — конкретные форматы запросов/ответов
3. **Стратегия скейлинга** — горизонтальный/вертикальный, лимиты, узкие места
4. **Мониторинг и observability** — что трекается, какие алерты, какой стек логов/метрик
5. **Воспроизводимость** — версионирование данных и моделей, experiment tracking

**Стандартные нотации архитектуры:** C4 (Context → Container → Component → Code), 4+1 view model, ERD, Data Flow Diagram. Любая из них — лучше списка логотипов.

Не применимо для чисто исследовательских работ без production-части — для них критерий помечается «н/п».

## 11. Воспроизводимость (сквозной критерий)

✅ Указаны все ключевые гиперпараметры (learning rate, batch size, epochs, dropout, seed, optimizer). Описано окружение (GPU/CPU, RAM, версии библиотек). Указан **бюджет ресурсов** (время обучения, общий compute). Код доступен (GitHub) или приложен Docker для запуска. Если работа с данными — данные опубликованы (HuggingFace, Zenodo) или приложена инструкция воспроизведения сбора. Для LLM-работ — зафиксированные промпты, **точная версия модели** (не просто «GPT-4», а `gpt-4-1106-preview`), температура (часто T=0), кэширование вызовов через трейсинг.
❌ Нет таблицы гиперпараметров. Нет seed — результаты невозможно повторить точно. Нет ссылки на код. Нет описания окружения. Не указано время обучения. Для LLM — не указана версия модели и температура.

Правило: эксперимент, который нельзя повторить, не является научным результатом. Industrial-работа без Docker / инструкции деплоя — не готова к продакшену.

**Дополнительно для русскоязычных LLM-экспериментов:**
- Проверена ли применимость токенайзера модели к русскому языку? (некоторые англоязычные tokenizer'ы выдают `<unk>` на кириллицу несмотря на заявленную мультиязычность)
- На каком языке промпт LLM-критика? (промпт на английском при русскоязычных данных — частая причина деградации Faithfulness/judge-метрик)
- Если используется sentence splitter — выбран ли русскоязычный (например `razdel` вместо `pysbd`)?

**Минимальный артефакт воспроизводимости:**
- Таблица гиперпараметров с grid search (если применялся)
- Версии всех ключевых библиотек (PyTorch, CUDA, фреймворки)
- Точное железо (модель GPU, RAM, число карт)
- Время обучения / inference в человеко-часах или GPU-часах
- Seed (или несколько seed'ов с std если делали)
- Ссылка на код (или указание, что код под NDA — с описанием альтернативного способа воспроизведения)

## 12. Раскрытие использования AI-инструментов (обязательное требование 2025/2026)

✅ Есть отдельный раздел / приложение с указанием: какие AI-инструменты использовались (ChatGPT, Claude, Copilot, Cursor, Gemini и др.), для каких задач (написание текста, генерация кода, анализ данных, поиск литературы, перевод), в каком объёме. Указано, что выводы, архитектурные решения, выбор методов и анализ результатов сделаны автором.
❌ Раздел отсутствует — это **этическое нарушение, эквивалентное плагиату** по правилам ИТМО 2025/2026 (см. семинар «ВКР без боли» #1).

Дополнительный тест: если **AI-часть можно убрать без существенных изменений в работе** — она не соответствует направлению «Искусственный интеллект» (см. также критерий 0).

Минимальный формат раскрытия:

> При подготовке работы использовались следующие AI-инструменты:
> - ChatGPT-4 / Claude 3.5 — редактура формулировок, проверка грамматики
> - GitHub Copilot — ускорение написания типового кода (boilerplate)
> - [AI-tool] — [для чего]
>
> Все архитектурные решения, выбор методов, проведение экспериментов и анализ результатов выполнены автором.

## Бонус. Внешняя валидация

Не входит в основной счёт критериев, но **сильно усиливает работу** и обязательно отмечается в шапке разбора:
- **Публикации** на конференциях / в журналах (Q1, ВАК, тезисы, постеры)
- **Пилот** с реальными пользователями + измеренные метрики (CSAT, NPS, retention, SUS)
- **Открытые артефакты:** датасет (HuggingFace, Zenodo), код (GitHub), модели (HF Hub)
- **Выступления:** митапы, школы, семинары, конференции
- **Письма поддержки** от заказчика, акт внедрения, интеграция в production
- **Гранты, премии, benchmark-результаты**

Сильные форматы внешней валидации:
- Публикации Q1/Q2 (NeurIPS, ICML, ACL, EMNLP, ICLR, CVPR, IEEE Access, ACM-журналы), включая воркшопы топовых конференций
- Пилот с реальными пользователями + измеренные метрики продукта (CSAT, NPS, retention, SUS, churn)
- Открытые артефакты на признанных платформах (HuggingFace, Zenodo, GitHub) с реальной adoption (downloads, stars, citations, contributions)
- Регуляторный/закрытый бенчмарк (NIST FRVT, IREX и аналоги) — особенно ценно для биометрии и регулируемых доменов
- Внедрение в production с подтверждением (акт внедрения, письмо заказчика, портал с реальными пользователями)
- Грант / премия / победа в конкурсе — внешняя экспертная оценка
- Portfolio из нескольких внедрений / пилотов одной методологии (особенно для methodology-papers)

Если внешней валидации нет вообще — это не блокер для защиты, но шансы на «отлично» снижаются. Если есть хотя бы одна форма — упоминай это как сильную сторону в начале разбора.

## Итоговая шкала

В счёт идут критерии 1–9 + 11 = 10 базовых критериев. Критерий 10 (System Design) учитывается для Industrial / production-работ. Критерии 0 и 12 — обязательные «ворота». Бонус — отдельно.

| Критериев ✅ из 10 (1–9 + 11) | Интерпретация |
|---|---|
| 9–10 | Сильная постановка — готова к защите |
| 7–8 | Есть ядро, нужна доработка по конкретным пунктам |
| 4–6 | Требует переработки |
| 0–3 | Собираем с нуля — Режим B |

> **Критические ворота:**
> - Критерий 0 (допуск: AI как центральный вклад) — без него работа не для программы «Искусственный интеллект».
> - Критерий 12 (раскрытие AI-инструментов) — без него работа не допускается к защите по правилам 2025/2026.
>
> Если хотя бы одни ворота не пройдены — независимо от остальных оценок начинай разбор с этой проблемы и формулируй её как блокер.

## Чеклист для Mode E — Обзор литературы

Используй при разборе related work / Главы 1.

| # | Проверка | Что значит ✅ | Что значит ❌ |
|---|---|---|---|
| E1 | Исследовательский вопрос | Сформулирован явно: тема → объект → предмет → вопрос | «Изучить область X» — это не вопрос |
| E2 | Ключевые термины поиска | Перечислены ru/en термины, синонимы, смежные области | Поиск только по русскому термину или только по основной фразе |
| E3 | Качество источников | Преимущественно Q1-конференции (NeurIPS, ICML, ACL, EMNLP, ICLR, CVPR), журналы Q1/Q2, рецензируемые статьи | Хабр, блоги, Medium, статьи без рецензирования как основные источники |
| E4 | Свежесть | Преимущественно 2020+ (если задача не историческая) | >70% источников старше 2018 |
| E5 | Метод снежного кома | Использован: важная статья → её цитирования → ещё статьи | Только первые результаты Google Scholar |
| E6 | Вердикт по каждому источнику | У каждого источника есть фраза «не подходит для нашей задачи, потому что [конкретно]» | Источники пересказаны, но не оценены применительно к задаче автора |
| E7 | Кластеризация и синтез | Источники сгруппированы по подходам / эпохам / школам, видна эволюция | Линейный список «авторы X, Y, Z сделали...» |
| E8 | Gap явно сформулирован | Конкретный пробел: «никто не сделал X для контекста Y» | «Область исследована недостаточно» — расплывчато |
| E9 | Связь gap → гипотеза | Сформулированный gap логически приводит к гипотезе работы | Гипотеза «висит в воздухе», обзор её не подготавливает |

**Антипаттерн:** «я уже построил решение — теперь подгоню обзор». Видно по тому, что обзор не имеет своей логики, источники подобраны под уже выбранный метод, gap натянут.

**Эталон обзора датасетов / методов:** для каждого из N существующих ресурсов даётся явный вердикт «не подходит, потому что [конкретное ограничение применительно к нашей задаче]». Например, при работе с русскоязычным доменным NLP: «датасет A — только английский; датасет B — нет терминологии нашего домена; датасет C — экстрактивные рефераты, а нам нужны абстрактивные». Gap из такого обзора **выводится** (а не постулируется), и гипотеза вытекает из gap логически.

## Чеклист для Mode F — Презентация защиты

Используй при разборе структуры презентации или готовых слайдов.

| # | Проверка | Что значит ✅ | Что значит ❌ |
|---|---|---|---|
| F1 | Проблема понятна за 2 минуты | На первых слайдах: кто пользователь, какой ущерб, почему сейчас | Контекст растянут на 5+ слайдов до сути проблемы |
| F2 | Конкуренты / аналоги | Есть слайд со сравнением аналогов и их ограничений | Свой подход показан в вакууме, без сравнения |
| F3 | Архитектура ≤ 2 слайдов | System Design занимает 1–2 слайда, остальное — не про архитектуру | Архитектура / стек растянуты на 4+ слайда логотипов |
| F4 | Цифры в результатах | На каждом слайде с результатом — конкретные цифры (метрики, дельты, сравнения с baseline) | «Система показывает хорошие результаты» без чисел |
| F5 | Слайд с ограничениями | Отдельный слайд: 3+ ограничений, явно названные | Ограничений нет, или одно предложение в выводе |
| F6 | Запасные слайды для Q&A | Подготовлены к 5–10 типичным вопросам (метрики, ablation, sample size, экономика) | Q&A не предусмотрено — ответы будут импровизироваться |
| F7 | Личный вклад выделен | Если работа в команде — явно сказано «автор отвечал за X, Y и Z» | «Мы сделали...» без разделения вклада |
| F8 | Trade-offs названы | На слайде с архитектурой / методом — почему выбрано именно так и от чего отказались | Решения показаны как единственно возможные |

**Главный принцип защиты:** *«Атакуй первым» — назови 2–3 слабых места работы до того, как их найдут эксперты. Это снижает напряжение Q&A и показывает зрелость.*

**Эталонные структуры презентации:**
- **Industrial-трек (12–15 слайдов):** проблематика → конкуренты → анализ их недостатков → преимущества решения → архитектура (1–2 слайда) → результаты → MVP-демонстрация → оценка бизнес-влияния → стоимость / экономика → ограничения → заключение и future work → запасные слайды для Q&A.
- **Science-трек:** Motivation → Task formulation → Related work / gap → Method → Experiments → Results → Limitations → Contributions and future work.
- **Startup-трек:** Problem → Customer → Solution → Market → Unit-economics → Traction / pilot → Competition / unfair advantage → Roadmap.
- **Education-трек:** Audience → Pain → Approach → Pilot design → Effect (с CI) → Limitations → Scalability.
