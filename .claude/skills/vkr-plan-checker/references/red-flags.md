# Топ ошибок в плане и постановке задачи (методологические red flags)

---

## Ошибки в описании проблемы

**1. Проблема подменяется решением**
❌ «Целью данной работы является разработка системы X»
✅ «Проблема — Y. Для её решения разрабатывается система X»

**2. Нет целевой аудитории и реального ущерба**
❌ «Задача актуальна для многих пользователей»
✅ «Врачи в клиниках первичного звена теряют 3–4 часа в день на ручную разметку. При усталости вероятность ошибки вырастает до 15%.»

Правило: проблема без ущерба — это не проблема, это наблюдение.

**3. Расплывчатая актуальность без «почему сейчас»**
❌ «Тема является актуальной и востребованной»
✅ «С 2024 года объём X вырос втрое, при этом существующие методы не масштабируются — разрыв нарастает»

**4. Ограничение существующих решений без конкретики**
❌ «Существующие методы недостаточно точны»
✅ «Метод X даёт точность Y на стандартных данных, но падает до Z на данных нашего типа — это показано в [ссылка]»

---

## Ошибки в гипотезе

**5. Гипотеза вообще отсутствует**
Симптом: во введении написано только «цель — разработать систему»
Починка: ответить на вопрос «что именно я пытаюсь доказать?»

**6. Гипотеза не измерима**
❌ «Система будет работать эффективно»
✅ «Точность > 90% на датасете X по метрике F1»

**7. Нет baseline в гипотезе**
❌ «Метод улучшит результаты»
✅ «Метод превзойдёт [конкретный baseline] по [метрика] на [условие]»

**8. Нет порога значимости**
❌ «Метод покажет лучшие результаты, чем baseline»
✅ «...не менее чем на 8 п.п. — что соответствует клинически значимому улучшению»

**9. Гипотеза и вывод не связаны**
Симптом: в заключении «цель достигнута, задачи выполнены» без явной ссылки на гипотезу
Починка: каждый пункт заключения — явный ответ на одну из гипотез («H1 подтверждена: ...»)

---

## Ошибки в структуре плана

**10. Задачи — процессы, не результаты**
❌ «Изучить существующие методы», «Провести эксперименты»
✅ «Провести сравнительный анализ методов A, B, C и выбрать наиболее подходящий», «Провести эксперимент на датасете D с baseline B по метрикам M1, M2»

**11. Валидация — маленький раздел в конце**
Симптом: «Глава 4 — тестирование» (одна строка, 5 страниц в итоге)
Починка: переосмыслить центральную главу как «проверку гипотезы», а не «разработку»

**12. Related work — реферат, а не аргумент**
❌ Источники перечислены и пересказаны. Нет вывода «почему существующего недостаточно».
✅ Каждый упомянутый метод/датасет/продукт получает одну фразу: «не подходит, потому что [конкретное ограничение применительно к данной задаче]». Обзор заканчивается явно сформулированным пробелом.

Правило: related work — это доказательство, что пробел реален. Не список того, что вы прочитали.

**13. Ограничения работы спрятаны или отсутствуют**
❌ Ограничений нет. Или: «работа имеет ограничения, которые будут устранены в будущем» — без конкретики.
✅ Отдельный раздел или явный абзац: какие данные/условия/метрики не покрыты, при каких сценариях метод может не работать.

Правило: лучшие ВКР программы выносят ограничения в отдельный раздел. Это усиливает доверие к работе.

**14. Импакт одноуровневый**
❌ Описан только один вид ценности: только метрики, или только бизнес-эффект.
✅ Импакт на нескольких уровнях: алгоритмический (метрики) + системный (production) + пользовательский (до/после) + научный (открытые данные/публикации).

## Ошибки для слепого рецензирования

**15. Текст понятен только автору**
❌ «В нашем проекте», «как обсуждалось с научруком», «данная система»
✅ Полное определение при первом упоминании, нет личных отсылок

**16. Аббревиатуры без расшифровки**
❌ «Система использует RAG с HNSW-индексом для BM25-ретривера»
✅ Расшифровать все аббревиатуры при первом использовании

**17. Личный вклад не выделен**
❌ Повсеместное «мы разработали / мы провели»
✅ «Автор данной работы отвечал за X; компоненты Y и Z реализованы другими участниками и рассматриваются только как контекст»

---

## Ошибка допуска

**18. AI-часть не является центральным вкладом**
❌ «Система реализована на Python с использованием ML-библиотек» — при этом вся новизна в бизнес-логике или UX.
✅ AI/ML — это то, что создаёт новизну или решает задачу, которую без него решить нельзя. Убери AI-часть — и работа перестаёт быть квалификационной для программы «Искусственный интеллект».

---

## Ошибки в работах с LLM/RAG

**19. RAG без раздельной оценки retrieval и generation**
❌ «Система даёт правильные ответы в 85% случаев» — непонятно, ретривер плохо ищет или LLM галлюцинирует.
✅ «Retrieval: precision@5 = 0.91, recall@5 = 0.87. Generation: faithfulness = 0.94, correctness = 0.88. Сквозная accuracy = 85%.»

**20. Нет описания chunking-стратегии и её обоснования**
❌ «Документы разбиваются на чанки по 512 токенов»
✅ «Используется sliding window с overlap 128 токенов. Выбор обоснован: при 256 токенах теряется контекст абзаца, при 1024 — падает precision retrieval на 12% из-за размытия семантики чанка.»

**21. LLM как чёрный ящик без мониторинга**
❌ Нет трейсинга вызовов LLM, нет оценки стоимости, нет мониторинга галлюцинаций в production.
✅ «Используется Langfuse для трейсинга каждого вызова: prompt, retrieved context, response, latency, cost. Мониторинг drift через Prometheus.»

**22. Baseline — только «без RAG» вместо честного сравнения**
❌ Сравнение только RAG vs. LLM без RAG.
✅ Сравнение с несколькими baseline: LLM без RAG, BM25 + LLM, dense retrieval + LLM, существующие коммерческие решения.

Правило для RAG-работ: покажи, что каждый компонент пайплайна вносит вклад. Ablation study: убери RAG → что изменится? Замени ретривер → что изменится? Замени LLM → что изменится?

## Ошибки в мультимодальных работах

**23. Мультимодальность без ablation study по модальностям**
❌ «Мы объединили текст, аудио и видео — и получили результат лучше, чем у baseline.»
✅ «Убрали аудио → wF1 упал на 0.26%. Убрали Action Units → wF1 упал на 3.04%. Визуальные признаки дали +0.93%. Каждая модальность вносит измеримый вклад.»

Правило: если используешь несколько модальностей — покажи вклад каждой. Без ablation непонятно, нужны ли все модальности или часть из них — шум.

**24. Fusion-стратегия не обоснована**
❌ «Признаки конкатенированы и поданы в классификатор.»
✅ «Сравнили early fusion (конкатенация), late fusion (ансамбль) и slot-fusion (обучаемые веса модальностей). Slot-fusion показал лучший результат, потому что позволяет модели самой определить значимость каждой модальности для конкретного примера.»

**25. Собственный датасет без описания процесса создания**
❌ «Мы собрали датасет из 5000 примеров.»
✅ «Датасет собран из источников A, B, C. Дедупликация: perceptual hash + cosine similarity на ResNet50 (порог 0.95). Аугментация: повороты ±10°, горизонтальное отражение, случайный crop. Разбиение: 80/20, стратифицированное по классам. Итого 12000 примеров, дисбаланс классов: 60/40 → oversampling минорного класса.»

**26. Fine-tuning foundation model без обоснования выбора**
❌ «Мы дообучили Qwen2.5-VL на наших данных.»
✅ «Выбрали Qwen2.5-VL-7B потому что: (1) поддерживает multi-image input, (2) 7B параметров — баланс качества и вычислительных затрат, (3) LoRA rank=16, alpha=16 — минимальное количество обучаемых параметров при сохранении качества. Сравнили с LLaVA 3.2 и GeoChat — Qwen показал лучший zero-shot на нашем домене.»

## Ошибки в NLP-работах и работах с датасетами

**27. Новый датасет без обоснования, почему существующие не подходят**
❌ «Мы создали новый датасет для задачи X.»
✅ «Проанализировали 11 существующих датасетов. Каждый получил вердикт: CNN/DailyMail — только английский, MLSUM — нет торговой терминологии, XSum — экстрактивные рефераты. Ни один не покрывает наш домен → необходим новый корпус.»

**28. Seq2seq модель без описания pipeline создания данных**
❌ «Мы обучили модель на парах (вход → выход).»
✅ «Pipeline: LaTeX-файлы из arXiv → удаление макросов (DeMacro) → конвертация в HTML (make4ht) → конвертация в Markdown (markdownify) → сопоставление с PDF по страницам (SIFT/ViT). Итого 300K пар. Описаны проблемы: 10% невалидных LaTeX, дубликаты, нестандартные пакеты.»

**29. Domain-specific модель без сравнения с general-purpose**
❌ «Мы использовали модель, предобученную на доменных данных.»
✅ «Domain-specific LM (предобученная на SO, 27B токенов) превзошла general-purpose модели сопоставимого размера (BERT-base, RoBERTa-base) на всех downstream-задачах. Преимущество: домен-специфичный токенизатор не теряет технические термины.»

**30. Работа с заказчиком без формализации требований**
❌ «Заказчик попросил сделать систему аналитики.»
✅ «Требования заказчика формализованы: (1) детекция объектов с mAP > 0.85 при FPS > 15, (2) интеграция с существующей CRM, (3) отчёты в формате PDF. Пилот: 4 недели, 50 пользователей, обратная связь через NPS.»

## Ошибки в CV-работах и работах с edge-деплоем

**31. Оптимизация модели без системного сравнения**
❌ «Мы квантизировали модель в INT8 и задеплоили на устройство.»
✅ «Сравнили FP32, FP16, INT8. F1: 0.953 → 0.951 → 0.951 (FP16 INT8 потеря < 0.2%). Скорость: 6530 → 28631 → 43211 кадров. Выбрали INT8 — потеря качества минимальна, скорость выросла в 6.6 раз.»

**32. Edge-деплой без обоснования выбора устройства**
❌ «Мы задеплоили модель на Raspberry Pi.»
✅ «Сравнили 3 устройства: Jetson Nano (0.5 TFLOPS, $250), Orange Pi 5 Pro (6 TOPS INT8, $67), Raspberry Pi 5 (0.015 TFLOPS, $46). Выбрали Orange Pi — лучшее соотношение INT8-производительности к цене для нашей задачи.»

**33. Pruning без анализа деградации**
❌ «Мы обрезали 30% параметров и модель всё ещё работает.»
✅ «Structured pruning (BatchNorm slimming): при 30% обрезке f1 сохраняется (0.953 → 0.953). При 50% — падение до 0.922. При 75% — до 0.901. Порог деградации — 40%. Unstructured pruning деградирует быстрее: уже при 10% f1 падает до 0.838.»

**34. Нет экономической оценки импакта для industrial-трека**
❌ «Система работает быстрее существующих аналогов.»
✅ «При снижении времени реагирования на 30% ожидаемое снижение ущерба = 3.57 млрд руб/год. Стоимость одного комплекта БПЛА + edge-устройство = 8 млн руб. ROI при масштабе > 50 устройств — менее 2 лет.»

## Ошибки в работах с инструментами / платформами

**35. User study без корректной статистики**
❌ «Мы провели опрос 10 человек — всем понравилось.»
✅ «User study: N=18, within-subjects design (каждый участник пробует оба интерфейса). Время: Scheirer-Ray-Hare test (p=0.05). Удовлетворённость: Wilcoxon signed-rank test по 6 Likert-шкалам (все p < 0.005). Контроль порядка: 2 группы (1D→2D и 2D→1D).»

Правило: если делаешь user study — опиши дизайн (within/between subjects), размер выборки, статистический тест и p-value. Для малых выборок (N < 30) — непараметрические тесты (Wilcoxon, Mann-Whitney, Scheirer-Ray-Hare), не t-test.

**36. Инструмент без обоснования проблемы данными**
❌ «Data scientists нуждаются в лучших инструментах для A/B-тестирования.»
✅ «Опрос 126 DS-специалистов: 74% не имели практики A/B до первой работы. 36% допускали ошибки в первый год. 67% назвали главной проблемой отсутствие практических тренажёров.»

**37. Бенчмарк / user study без воспроизводимости**
❌ «Мы провели эксперимент и получили результаты.»
✅ «Docker-окружение для воспроизведения бенчмарка. Все execution traces исходного датасета воспроизведены в идентичных условиях. Промпты, температура (T=0) и тест-кейсы зафиксированы — эксперимент воспроизводим.»

## Ошибки в работах с anomaly detection и мультиагентными системами

**38. Anomaly detection без сравнения supervised / semi-supervised / unsupervised**
❌ «Наш метод обнаруживает 85% аномалий.»
✅ «Три уровня сравнения: unsupervised (PCA+Kmeans: TPR=0.36), semi-supervised с 5% разметки (наш метод: TPR=0.69), supervised с 100% разметки (TPR=0.84). Наш метод достигает 82% качества supervised при 5% стоимости разметки.»

Правило: для задач с дорогой разметкой (промышленность, медицина) — покажи trade-off между объёмом разметки и качеством. Active learning = ценность работы.

**39. Мультиагентный pipeline без ablation по компонентам**
❌ «Система состоит из STT, LLM, TTS и анимации и работает.»
✅ «Сравнили 6 TTS-систем по 3 осям (MOS, latency, VRAM): высококачественная (MOS 4.6, latency 6–7 сек), компромиссная (MOS 4.2, 7–9 сек), быстрая (MOS 3.1, 0.8 сек) и т.д. Выбрана высококачественная — лучший MOS при приемлемой latency. LLM: сравнили 7B vs 12B — 12B на 34% точнее, но в 2 раза медленнее; выбрали 12B с 4-битной квантизацией.»

**40. Соревнование / хакатон как единственная валидация**
❌ «Мы заняли 1 место на хакатоне — значит, система работает.»
✅ «Соревнование / публичное мероприятие — одна форма внешней валидации. Но также: A/B-тест нескольких персон агента (десятки участников, тысячи сообщений), метрики удержания, время автономной работы, uptime — измеренные параметры real-time системы.»

## Ошибки этики и оформления (обязательное с 2025/2026)

**41. Раздел о раскрытии AI-инструментов отсутствует**
❌ В работе использовались ChatGPT/Claude/Copilot, но об этом в тексте ни слова.
✅ Отдельный раздел / приложение: «При подготовке работы использовались следующие AI-инструменты: ChatGPT-4 — редактура формулировок; GitHub Copilot — ускорение написания типового кода; Claude — поиск литературы и проверка грамматики. Все архитектурные решения, выбор методов, проведение экспериментов и анализ результатов выполнены автором.»

Правило: с 2025/2026 учебного года ИТМО требует обязательного раскрытия. **Нераскрытие приравнивается к плагиату** — это блокер допуска к защите. Подробно разбиралось на семинаре «ВКР без боли» #1.

**42. AI-часть можно убрать без существенных изменений**
❌ Работа называется «AI-система для X», но ML-часть — это вызов готового API без файнтюнинга, без собственных данных, без анализа метрик. Вся новизна — в UX или бизнес-логике.
✅ AI/ML создаёт уникальную возможность, которую нельзя получить иначе: персонализация на собственных данных, файнтюнинг под доменную задачу, новый бенчмарк, новая архитектура fusion и т.д. Тест: «убери AI — работа всё ещё квалификационная для программы "Искусственный интеллект"?» Если да — AI-часть нужно усилить или работа не для этой программы (см. также критерий 0 чеклиста).

## Ошибки воспроизводимости

**43. Эксперимент описан без гиперпараметров и seed**
❌ «Мы обучили модель на наших данных и получили accuracy 0.87.»
✅ «Обучение: AdamW (lr=2×10⁻⁴, weight_decay=0.01), batch=32, epochs=10, dropout=0.1, seed=42. Окружение: NVIDIA A100 40GB, PyTorch 2.1, CUDA 12.1. Код: github.com/.../repo. Гиперпараметры подобраны grid search по сетке {lr ∈ [1e-5, 1e-3], batch ∈ [16, 32, 64]}.»

Правило: эксперимент без зафиксированных гиперпараметров и seed нельзя воспроизвести точно — это снижает доверие к результатам.

**44. LLM-эксперимент без указания версии модели и температуры**
❌ «Использовали GPT-4 для оценки.»
✅ «GPT-4-turbo (`gpt-4-1106-preview`), температура T=0, max_tokens=512, system prompt зафиксирован (Приложение Б). Дата проведения: 12–18 марта 2025. Кэширование ответов через Langfuse — все вызовы воспроизводимы.»

Правило: LLM-провайдеры регулярно меняют модели за тем же именем. Без точной версии и температуры эксперимент не воспроизводим даже автором через 3 месяца.

**45. Нет ссылки на код / данные / Docker**
❌ «Реализация выполнена на Python.»
✅ «Код: github.com/.../repo (MIT). Датасет: huggingface.co/.../dataset. Docker Compose поднимает весь стек одной командой `docker compose up`. README содержит инструкции воспроизведения экспериментов из таблицы 4.2.»

## Ошибки System Design

**46. Стек вместо архитектуры**
❌ Слайд / абзац: «Использовались Docker, Kubernetes, Kafka, Airflow, FastAPI, PostgreSQL, Redis, Prometheus, Grafana» — список логотипов / названий без обоснования.
✅ Диаграмма (C4 / Data Flow / ERD) + по каждому компоненту: какую роль играет, какие альтернативы рассмотрены, почему выбран именно этот, какие нефункциональные требования он закрывает (latency, throughput, отказоустойчивость).

Правило: диаграмма с логотипами — не архитектура. Архитектура — это **цепочка решений с обоснованием**.

**47. Архитектура без trade-offs**
❌ «Выбрали PostgreSQL.»
✅ «Сравнили PostgreSQL, MongoDB, ClickHouse. Выбрали PostgreSQL: (1) транзакционность ACID — критично для финансовых данных, (2) JSONB для гибкости полу-структурированных полей, (3) расширение pgvector закрывает задачу хранения эмбеддингов без отдельного векторного хранилища. Trade-off: проигрываем ClickHouse в 5–10× на аналитических запросах — но они выполняются ночными батчами и не требуют real-time.»

**48. Нефункциональные требования не указаны**
❌ «Система обрабатывает запросы пользователей.»
✅ «НФТ: latency p95 < 500 мс при нагрузке 200 RPS, доступность 99.5%, RTO 15 мин, RPO 5 мин, горизонтальная масштабируемость до 10 инстансов без переписывания кода. Каждое НФТ обосновано продуктовой метрикой (CSAT падает на 18% при latency > 1с — поэтому p95 < 500 мс).»

## Ошибки обзора литературы

**49. Related work как реферат, без вердиктов**
❌ «Авторы X предложили метод Y. Авторы A предложили метод B. Авторы C предложили метод D.»
✅ «Метод Y (X et al., 2022) достигает F1=0.84 на CONLL, но деградирует до 0.61 на доменных текстах с code-mixed терминологией — что является блокером для нашей задачи. Метод B (A et al., 2023) требует > 100K размеченных пар, недоступных в нашем домене. Метод D (C et al., 2024) ближе всего к нашей задаче, но не публикует код и не включает русский язык в эксперименты.»

Правило: каждый источник = вердикт «почему не подходит для нашей задачи». Обзор — это доказательство существования gap, а не список того, что вы прочитали.

**50. Источники только с Хабра / Medium / блогов**
❌ Основные источники — посты на Хабре, статьи на Medium, блоги вендоров.
✅ Преимущественно рецензируемые публикации: Q1-конференции (NeurIPS, ICML, ACL, EMNLP, ICLR, CVPR, IEEE), журналы Q1/Q2, arXiv с указанием статуса (peer-reviewed / pre-print). Блоги допустимы как иллюстрация production-практик, но не как основа.

**51. Gap «область исследована недостаточно»**
❌ «Несмотря на множество работ по теме X, область остаётся недостаточно исследованной.»
✅ «Из 14 проанализированных работ ни одна не покрывает русскоязычный контекст в условиях малой разметки (< 1000 примеров). Работа [Y, 2024] ближе всего, но обучена только на английском и не выложила веса. Это и есть gap — на нём строится гипотеза.»

## Ошибки презентации защиты

**52. Архитектура растянута на 4–5 слайдов**
❌ Слайд «Архитектура»: 5 слайдов с разными уровнями детализации, каждый с логотипами стека.
✅ 1–2 слайда: высокоуровневая диаграмма (C4 Context / Container) + ключевая идея архитектуры в одной фразе. Детализация — в backup-слайдах для Q&A.

Правило: System Design занимает 1–2 слайда максимум. Если хочется больше — это backup для Q&A.

**53. Скрытые ограничения вместо «атаки первым»**
❌ Ограничений в презентации нет — рецензент находит их сам и формулирует жёстко.
✅ Отдельный слайд «Ограничения и Future work» с 3+ конкретными boundary conditions. Подача в речи: «До того, как вы зададите вопросы, я хочу сам отметить три места, где работа имеет ограничения...» Это снижает напряжение Q&A.

**54. Результаты без сравнения**
❌ «Accuracy: 0.87.»
✅ «Accuracy: 0.87 vs baseline (BERT-base zero-shot) 0.62 vs SOTA (mBART-50) 0.83 на нашем доменном тестовом наборе из 1200 примеров. Прирост против SOTA — 4 п.п. (statистически значимо, McNemar p < 0.01).»

## Ошибки экспериментального дизайна

**55. Cherry-picked примеры без метода отбора**
❌ Раздел «Качественный анализ» показывает 4 удачных ответа модели без указания, как они выбраны, сколько было неудачных, какова их доля.
✅ «Из 200 тестовых примеров отобрано по 3 представителя каждого из 4 классов ошибок (выборка по доле). Доля удачных ответов на полном тесте — 73%, доля провалов разбита по типам в таблице N».

**56. Метрики без CI / σ / стат-теста**
❌ «Accuracy 0.64 vs 0.58 — наш метод лучше».
✅ «Accuracy 0.64 ± 0.012 (95% CI) vs 0.58 ± 0.014, разница статистически значима (McNemar p < 0.01, n=2000)». Для научного трека — обязательно std по ≥3 seed'ам.

**57. Противоречивая воспроизводимость**
❌ Гиперпараметры даны двумя разными списками в разных частях работы, значения местами расходятся (max_steps=30 vs num_train_epochs=10, lr 1e-4 vs 2e-4).
✅ Один источник истины — таблица в приложении или явная сводка с обоснованием расхождений (если разные стадии обучения имеют разные lr — это явно прописано).

**58. «Упомянул в обзоре, но не сравнил»**
❌ В обзоре подробно разобраны 8 моделей, в экспериментах сравнение только с 2.
✅ Каждая упомянутая в обзоре конкурирующая модель либо появляется в экспериментах как baseline, либо явно указано «не использовалось как baseline, потому что [нет открытых весов / другой домен / устарело]».

**59. Цифры в Conclusion, не подтверждённые в Results**
❌ В заключении «снижение времени на 22.7%, 40.5% меньше ошибок, 34% снижение когнитивной нагрузки», но в Results только одна цифра 18% и Likert-шкалы.
✅ Все цифры из Conclusion подкреплены конкретной таблицей или абзацем в Results с тем же числом.

**60. Ablation проводится последовательно, а не контролируемо**
❌ В серии экспериментов одновременно меняются архитектура + данные + loss — невозможно атрибутировать улучшение конкретному компоненту.
✅ Контролируемый ablation: одна переменная за раз. Если меняли несколько — отдельные таблицы изоморфного факторного дизайна.

**61. Composite metric / scoring без обоснования весов**
❌ `score = 0.3·M₁ + 0.3·M₂ + 0.4·M₃` без объяснения, почему именно эти веса.
✅ Веса либо подобраны grid search (с указанием диапазона), либо обоснованы доменом («NER-F1 важнее, потому что фактическая корректность критичнее лексического совпадения для нашей задачи»), либо приведён анализ чувствительности результата к выбору весов.

**62. Single LLM-judge без cross-validation вторым**
❌ Качество результата оценивается одним LLM-судьёй (например, GPT-4) без проверки альтернативной моделью.
✅ Два независимых LLM-судьи разных провайдеров + сверка их разногласий. Если расхождения значительные — это отдельная находка (калибровка судьи).

**63. Negative sampling только случайный (для metric learning)**
❌ Триплеты (anchor, positive, random_negative) без hard negative mining.
✅ Hard negative mining через текстовое сходство (TF-IDF + порог) или semi-hard mining. Случайные негативы приводят к коллапсу эмбеддингов.

**64. Demo как «снятая» валидация**
❌ Веб-демо или Telegram-бот приведены как функциональный артефакт без user-study, без количественной оценки UX, без метрик использования.
✅ Demo как артефакт + отдельные **метрики** (число пользователей / запросов / retention / CSAT / NPS / SUS). Различай demo-as-artifact и demo-as-validation.

**65. Datasets без anti-leak проверки**
❌ Абстрактивный датасет (суммаризация / генерация) использован без проверки, есть ли в reference-выходах дословные предложения из входа.
✅ Автоматический фильтр на extractive leak с указанием доли (например, «0.5% пар содержат дословные предложения, исключены»).

**66. Latency не определена как acceptance criterion до экспериментов**
❌ В конце работы выясняется, что система отвечает 60–300 сек при 3.6 LLM-вызовов на запрос — критерий «приемлемой задержки» не задавался.
✅ В разделе требований чётко: «Целевая end-to-end latency p95 < X сек». Все эксперименты сравниваются с этим порогом.

**67. Использование отозванных датасетов без указания**
❌ В работе используется датасет, который был отозван по этическим причинам (например, MS-Celeb-1M), без упоминания этого факта.
✅ Указано «датасет X был отозван авторами в [год]; в нашей работе использовалось доступное подмножество с обоснованием [...]».

**68. NDA как оправдание невоспроизводимости без компенсации**
❌ «Датасет под NDA, опубликовать не можем» — и больше никаких компенсирующих мер.
✅ NDA признан явно + одна из компенсаций: synthetic-аналог с тем же распределением, описание схемы и статистики, открытый бенчмарк той же природы для частичной воспроизводимости, anonymized дамп.

**69. Cherry-picking бенчмарков**
❌ Отчёт только о тех бенчмарках, где предложенный метод выигрывает.
✅ Полная таблица по всем релевантным бенчмаркам, включая те, где метод проигрывает или показывает паритет — с обсуждением причин.

**70. LLM-эксперимент без точной версии модели и температуры**
❌ «Использовали GPT-4 для оценки».
✅ Точное имя версии (например, `gpt-4-1106-preview`), температура (T=0 для воспроизводимости), max_tokens, system prompt в приложении, дата проведения, кэширование вызовов.

**71. Промпт LLM-критика на чужом языке**
❌ Промпт критика на английском при русскоязычных оцениваемых данных — деградация Faithfulness и judge-метрик.
✅ Промпт критика **на том же языке**, что и оцениваемые данные. Для русскоязычных пайплайнов — проверка применимости токенайзера к кириллице (некоторые «мультиязычные» модели выдают `<unk>` на русский). Для sentence splitter — `razdel` вместо `pysbd`.

**72. Ablation без строки «+N эпох baseline»**
❌ Метод X улучшает метрику на +2.3% при +50% времени обучения — но непонятно, не выиграл ли просто за счёт большего compute.
✅ В таблице ablation отдельная строка: «baseline + дополнительные N эпох без других изменений» — это отделяет реальный выигрыш метода от эффекта compute.

**73. User study с experimenter bias**
❌ User study проводит сам автор, прекрасно знающий, что новый интерфейс «должен» победить — нет blinding или независимого фасилитатора.
✅ Независимый фасилитатор / blind условия (если возможно) / счётчик времени и логи действий, не зависящие от субъективной оценки автора.

**74. Counterbalancing нарушен постфактумным исключением участников**
❌ 22 участника распределены по 2 группам, потом «3 исключены для равенства групп» — рандомизация нарушена.
✅ Если кого-то исключаем — критерий исключения сформулирован **до** анализа результатов и применяется механически.

**75. Отсутствие baseline-агента в собственном бенчмарке**
❌ В новом бенчмарке для генерации/предсказания нет тривиальных baseline (`random`, `repeat-last`, `most-frequent`) — непонятно, насколько метрика 0.13 выше случайного.
✅ Минимум 2 тривиальных baseline в дополнение к LLM/ML-методам.

**76. Утверждения о RT с тяжёлыми операциями**
❌ Метод заявлен как real-time, но содержит DBSCAN / pairwise сравнение всех со всеми — на больших данных не работает.
✅ Если real-time — продемонстрировано на целевой нагрузке в production-условиях, с указанием инкрементального workaround в ограничениях.

**77. Биометрия / классификация с асимметричной стоимостью без operating point**
❌ Только ROC-AUC = 0.87 без указания, при каком FAR работает система.
✅ Указан конкретный operating point: TAR@FAR=0.01% или FP-rate ≤ X% для anti-cheat / T&S задач.

**78. Self-hosted LLM без расчёта памяти**
❌ «Развернули модель на собственном сервере» без указания GPU, VRAM, формы квантизации.
✅ Расчёт памяти (формула «параметры × байт/параметр» с поправкой на KV-cache), указание GPU и квантизации (AWQ / GPTQ / BnB).

**79. Композитная метрика валидируется только корреляцией с одной из её же компонент**
❌ Композитная метрика валидируется через её корреляцию с компонентой M₁, которая входит в её формулу с весом 0.6 — корреляция тривиальна.
✅ Валидация на внешнем критерии (экспертная разметка, downstream-метрика, человеческая оценка).
