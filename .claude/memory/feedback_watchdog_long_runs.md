При запуске любого long-running процесса (LLM-агенты, обучение PPO, генерация preference matrix, batch-вызовы API) **обязательно ставить активный watchdog**, не полагаться только на ожидание файла.

**Why:** пользователь несколько раз поймал ситуацию, когда я ждал файл результата, а процесс молча висел (CPU=0, нулевой stdout). Дедлок проявлялся как «бесконечно гонится». Это потеря 30-60 минут пользовательского времени, и происходило неоднократно. Цитата: «уже блять который раз ловлю ситуацию, когда мы хуй пойми чего ждем, а потом оказывается, что система зависла».

**How to apply:**
- Не использовать `until [ -f ... ]; do sleep ...` как единственный механизм ожидания.
- Вместо этого — Monitor c grep на `Running policy|elapsed|Traceback|Error|FAILED` или Bash watchdog: проверять `ps -o %cpu,etime,time` каждые 5-10 минут. Если CPU=0% и elapsed > 10 минут без новых строк stdout → процесс мёртв, kill и расследование.
- При запуске нескольких политик подряд проверять ход после каждой (event "Running policy: X" пришёл, "elapsed" пришёл — ОК; иначе — поймал зависание).
- В коде, который делает много LLM-вызовов: `timeout=60.0`, `max_retries=2` обязательны. Между фазами (политиками, итерациями обучения) пересоздавать `AsyncOpenAI` client — старый может залипнуть на connection pool после 800+ запросов.
- Активно говорить пользователю: «процесс не показал прогресса 10 минут, проверяю / убиваю» — не молчать в надежде что он сам отвиснет.
- При повторном зависании — не перезапускать те же параметры. Снизить concurrency, добавить логирование, сделать минимальный repro (5-10 итераций).

**Маркер реальной активности:** stdout продолжает расти. Если `wc -c output_file` не меняется 2-3 минуты на LLM-нагрузке — это уже повод тревожиться.
