Перед запуском любого прогона, чьи числа войдут в таблицы Главы 4 или будут использоваться для интерпретации результатов, **в явном виде проговорить пользователю ключевые параметры** и получить подтверждение. Не использовать дефолты argparse молча.

Минимальный список параметров для валидации:
- `--user-compliance` (или `--calibrated-compliance`)
- `--w-fame`
- `--relevance` (cosine/learned)
- `--K`, `--tau`, `--lambda-overflow`
- `--seeds` (сколько и какие)
- `--personas` (какой набор и сколько)

**Why:** на сессии 02.05.2026 я гонял Demo Day с `--user-compliance 1.0` (дефолт скрипта) во всех прогонах, не уточнив у пользователя. Это исказило интерпретацию результатов: при compliance=1.0 capacity-aware политика тривиально даёт `OF≈0`, потому что пользователь физически не может пойти в неотрекомендованный зал. Реалистичное значение по калибровке Meetup-данных = 0.717 (compliant) или калиброванная трёх-типная модель. Несколько часов обсуждения шло на «инфлированных» цифрах, которые не отражают защитимый сценарий.

**How to apply:**
- Перед прогоном таблицы политик распечатать текущие значения параметров и спросить «такие параметры?»
- Если пользователь не отвечал по конкретному параметру явно — спросить, не подставлять дефолт.
- Особое внимание: compliance, w_fame, relevance — эти три почти всегда содержательно меняют интерпретацию.
