**Правило:** для любого скрипта, который идёт дольше ~30 секунд (LLM-прогоны, обучение, симуляции, валидации, парсинг датасетов), **обязательно** прикручивать `tqdm` (или эквивалентный progress bar с ETA) на главный цикл. Не угадывать время устно («ждать 3-4 минуты»).

**Why:** пользователь не должен полагаться на мои оценки — они часто врут (как по затратам, так и по времени). Видимый ETA в stdout избавляет от вопросов «сколько ещё?», «как успехи?» и от риска фоновых зависаний без диагностики.

**How to apply:**
- В новом скрипте: `from tqdm import tqdm` + `for x in tqdm(items, desc="название"):`.
- Для async — `tqdm.asyncio.tqdm` или `tqdm.gather`.
- Если уже есть скрипт без tqdm — добавить **до** запуска (5 строк).
- В долгих прогонах с per-slot/per-batch print — заменить на tqdm с `set_postfix({"cost": f"${cost:.2f}", "OF": of_running})`.
- При запуске в фоне через Bash — `PYTHONUNBUFFERED=1` и stderr→stdout, чтобы tqdm-строка попадала в логи.
- Перепроверять через `Read` файл-output: если tqdm-строки давно не растут → зависание, kill.

**Связано с:** `feedback_long_runs_visible_progress.md`, `feedback_watchdog_long_runs.md`. Этот файл — конкретное усиление: «прогресс должен быть с ETA, не просто print раз в N итераций».
