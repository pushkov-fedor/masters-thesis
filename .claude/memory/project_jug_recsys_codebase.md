## ML-часть проекта (стартовая точка для ВКР)

### Что реально делает рексис jug-rec-sys

1. Пользователь в Telegram-тиндере лайкает / дизлайкает доклады.
2. Positive-лайки эмбеддятся (эмбеддер ~OpenAI-совместимый, вектор **1536-dim**).
3. Берётся **средний вектор** по positive_points (`strategy='average_vector'`).
4. По нему делается cosine search в **Qdrant** (коллекция `heisenbug_2026`, 34 доклада).
5. Возвращается top-N похожих.
6. Cold start → случайный вектор.

Это всё. Никакого ранжирования поверх, никаких ограничений (зал/время), никакого обучения, никаких альтернативных стратегий pooling.

### Ключевой код
- `recommendation-engine/src/services/recommendation_service.py:34-44` — рекомендательная логика.
- `recommendation-engine/src/core/qdrant.py` — обёртка над Qdrant Client.

### Чего в ML-части НЕТ (точки роста для ВКР)

- **Никакой evaluation:** ни NDCG, ни MRR, ни recall@K, ни корреляции с feedback.
- **Никаких baselines** (random, popularity, rule-based).
- **Сравнения эмбеддеров** не проводилось.
- **Ablation studies** нет.
- **Strategy='average_vector'** — единственная.
- **Нет учёта temporal/spatial constraints** — именно то, что Марк хочет видеть (балансировка залов, вероятностная модель посещения).
