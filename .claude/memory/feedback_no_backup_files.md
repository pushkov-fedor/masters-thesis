**НЕ СРАТЬ ФАЙЛАМИ.** Дословно — пользователь несколько раз прямо так сказал: «не сри файлами», «молю хватит срать файлами», «можешь в память так и записать: не СРАТЬ файлами».

Не делать:
- `cp file.json file.json.bak` перед редактированием.
- `file_OLD.json`, `file_OLD_<reason>.json.bak`.
- `script_v2.py`, `script_new.py`, `script_final.py` рядом со старым.
- `report_final.md`, `report_final_v3.md`, `report_actual.md`.
- Дубликаты «на всякий случай».
- Создавать новый скрипт, когда можно расширить существующий флагом.

**Why:**
- Проект под git — для отката `git checkout`/`git stash`/`git revert`. Бэкапы рядом не нужны.
- Хлам ломает ls/grep, попадает в эмбеддер случайно при индексации `data/`, и пользователь вынужден чистить вручную.
- Это раздражает пользователя сильнее, чем кажется со стороны.

**How to apply:**
- Прежде чем создать новый файл — спроси: можно ли расширить существующий? (флаг, параметр, ветка в коде).
- Перед `cp X X.bak` — стоп. Это под git? Если да — никаких бэкапов.
- Если файл вне git и правка реально рискованная — спросить пользователя, не делать `.bak` молча.
- При уборке проектов — активно искать и удалять `*.bak`, `*_OLD*`, `*.backup`, `*_old*`, `*_v2*` (если v2 не настоящий следующий релиз).
- При рефакторинге скриптов — переименовывать существующий, не плодить параллельный.
