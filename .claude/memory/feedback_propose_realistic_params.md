Если для прогона есть **более реалистичная** альтернатива дефолтному параметру (калиброванное значение из реальных данных, экспериментально обоснованное число из литературы и т.п.) — **явно предложить её пользователю**, не оставлять выбор за пользователем по умолчанию.

Конкретные пары «дефолт vs реалистичное» для текущей работы:
- `--user-compliance 1.0` (дефолт, нереалистичный потолок) **vs** `--calibrated-compliance` (трёх-типная модель 71.7/21.3/7.0, откалибрована на Meetup RSVPs).
- `--w-fame 0.0` (без star-эффекта) **vs** `--w-fame 0.3` (умеренный star-эффект, проверен в sensitivity-анализе).
- `--relevance cosine` (Pearson 0.36) **vs** `--relevance learned` (Pearson 0.79, **только для Mobius**, на Demo Day learned-модель не обучена).
- `--K 3` (макс) — корректно, K_t тогда зависит от слота.

**Why:** на сессии 02.05.2026 пользователь явно подсветил, что я молча использовал `--user-compliance 1.0` (дефолт скрипта), хотя реалистичное значение было известно (calibrated compliance уже реализована и откалибрована на Meetup RSVPs). Это исказило интерпретацию главных результатов — `Capacity-aware OF≈0` оказался тривиальным потолком из-за нереалистичного режима. Часы обсуждения шли на инфлированных цифрах.

**How to apply:**
- Перед прогоном явно сказать «дефолт X, но более реалистичное значение Y, потому что Z».
- Не использовать молча менее реалистичный дефолт — это активный выбор «понизить достоверность ради удобства», который должен быть осознанным.
- Особенно для compliance (есть явная калибровка), w_fame (есть sensitivity), relevance (есть learned-альтернатива).
