12 мая 2026 (накануне предзащиты) проведён повторный исчерпывающий поиск публичных датасетов, на которых можно было бы калибровать функцию полезности участника `U(t) = w_rel·rel + w_rec·1{t∈recs} + w_gossip·gossip`.

**Поиск выполнялся через general-purpose subagent с обязательной инспекцией данных** (не верить названиям и описаниям, скачивать и открывать). Бюджет 20 минут, использовано 14.

**Проверенные источники** (~30 за 14 минут): Kaggle, HuggingFace Datasets, Papers with Code, SocioPatterns, CRAWDAD, Internet Archive, arXiv, GitHub topic searches, conference-scheduling literature (Vangerven EJOR 2018, JORS 2024, Quesnelle, Rahim, PATAT), Sched.com, Queue Times API, ticketing platforms, RecSys datasets, smart card / transit data, music festival sensing, museum trajectory data, theme park data, online news recommendation (MIND/EB-NeRD).

**Ближайший кандидат — MAPSP/Vangerven** (EJOR 2018): 78 участников реальной академической конференции, параллельные сессии. Отвергнут потому что: (а) stated preferences опросником, не revealed choice; (б) capacity не публикуется; (в) дамп предпочтений не выложен публично.

**Главный вывод:** одновременное наличие трёх свойств — индивидуального выбора, параллельных альтернатив в один слот, сигнала ограниченной вместимости — в публичных датасетах не зафиксировано. Это **структурное свойство домена**, не дефект поиска. RFID/Bluetooth — face-to-face без привязки к сессиям; event-RSVP — независимые события без capacity; conference scheduling literature — либо синтетические preferences, либо stated. Online news recommendation — есть impression+click, но другой жанр без capacity.

**Дополнительная находка по Meetup** (на инспекции при попытке частичной валидации): `meetup_users.json` содержит профили вида `"Meetup user 30197, city=Amsterdam, country=nl"` — без интересов и демографии. Эмбеддинг через любой энкодер на таком профиле даёт сигнал только о городе. Это **второй независимый стопер** к Meetup как датасету для валидации формулы релевантности (первый — leakage через memberships в B1, уже в стоп-листе PROJECT_STATUS §5).

**Why:** комиссия защиты задаст вопрос «вы что, не искали данные?». Аргумент «искали один раз при подготовке работы» слабее, чем «искали два раза, включая исчерпывающий повторный поиск накануне с инспекцией данных, и подтвердили структурный отказ».

**How to apply:**
- В Q6 готовых ответов `presentation/speech.md` записан конкретный список проверенных доменов и причин отвержения. Использовать как опору устного ответа.
- НЕ запускать новый поиск в будущих сессиях без явного запроса — это полностью проработано.
- При вопросе об конкретном датасете — называть Vangerven как ближайший негативный пример, не SocioPatterns / Hangtime.
- В тексте речи и слайдах НЕ упоминать B1/accuracy@1=0.918/leakage — это в стоп-листе работы. Защита идёт через косвенные методы (граничная верификация Сарджента + два независимых симулятора).
