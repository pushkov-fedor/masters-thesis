## Главные выводы

### 1. Capacity-aware политики стабильно дают **0%** overflow на choice-слотах
По всему диапазону τ ∈ [0.1, 2.0], λ ∈ [0, 5], K ∈ {1, 2, 3} — это структурное свойство, не результат подгонки параметров. Цена — 2.5% утилитарности относительно Cosine (0.347 vs 0.356).

### 2. Главный вывод устойчив к смене relevance signal
- На **cosine relevance**: Capacity-aware OF=0.000, Cosine OF=0.389; Δutility=2.7%.
- На **learned relevance** (HistGradientBoosting на 12K LLM оценок): Capacity-aware OF=0.000, Cosine OF=0.306; Δutility=2.5%.

Это устраняет циркулярное возражение «оптимизируете cosine, валидируете cosine» — теперь у нас есть независимая модель предпочтений, обученная на сторонних оценках LLM.

### 3. Constrained-PPO — обучаемая политика — даёт промежуточный результат
Overflow 0.283 (лучше Cosine 0.306, хуже Capacity-aware 0.000), Utility 0.351 (близко к Cosine 0.356). Это показывает: RL с двойственной оптимизацией λ умеет балансировать, но не догоняет жёсткое action masking. **Содержательный вывод для главы 4:** rule-based политика остаётся наилучшей в режиме без сложной структуры состояния.

### 4. LLM-ranker без state-awareness — содержательный негативный результат
Overflow 0.344 (хуже Cosine), Utility 0.339 (между Random и Cosine). Чисто семантическое ранжирование LLM не лучше эмбеддингового косинуса; для решения задачи ограничения вместимости нужна явная информация о состоянии.

Это указывает на необходимость scale-up до 200-300 агентов для воспроизведения congestion-эффектов, что выходит за бюджет данной сессии.

## Стадия 6 — LLM-as-judge: завершено

Судья: Claude Sonnet 4.6. 1800 pairwise сравнений (15 пар политик × 20 персон × 6 слотов), стоимость $8.46, ~5 минут.

**Bradley-Terry rating (выше = судья чаще предпочитает выдачу этой политики):**

| Политика | BT-рейтинг |
|---|---|
| **LLM-ranker** | **1.331** |
| Capacity-aware MMR | 1.083 |
| Capacity-aware | 1.059 |
| MMR | 0.968 |
| Cosine | 0.953 |
| Random | 0.606 |

**Содержательное наблюдение для главы 4.** Судья (frontier-модель Sonnet 4.6) систематически предпочитает выдачи LLM-ranker'a, несмотря на то, что по системным метрикам (overflow, variance) LLM-ranker — одна из худших. Это **существенное расхождение** между субъективным качеством рекомендаций и системными свойствами:

- При оценке отдельных пар (judge view): LLM-ranker лучший — он умеет красиво подать выдачу, даёт текстово согласованные рекомендации.
- При системной оценке (overflow, capacity): LLM-ranker провальный — он создаёт переполнения, концентрируя пользователей в популярных залах.

Capacity-aware политики занимают 2-3 место по judge view (BT=1.06-1.08), что снимает возражение «вы уродуете релевантность ради метрики». Subjective quality они не уродуют, но системные метрики при этом сильно лучше.

Это центральный аргумент для защиты: **система должна оптимизировать не только субъективное качество (где LLM выигрывает), но и системные свойства (где capacity-aware выигрывает) — иначе теряется значительная доля пользовательского опыта (40+% переполнений vs 0%) при минимальном выигрыше в восприятии (BT 1.33 vs 1.06 = 25% в субъективной мере, но это легко компенсируется отсутствием физического дискомфорта от переполнения).**

## Что не успели — и что с этим делать

1. **Полный agent validation на 200+ агентах для воспроизведения overflow.** Текущие 50 агентов не нагружают залы. Решение: либо описать как ограничение и показать sim-to-sim gap по skip rate / utility, либо запустить 200 агентов отдельно (~$40, ~3 часа).
2. **PPO improvement.** Текущие 200K steps дают 0.283 overflow. Возможно, 1M steps + tuning гиперпараметров довели бы PPO до 0.05-0.1 overflow.
3. **Калибровка персон под реальную аудиторию JUG.** 300 LLM-персон не калиброваны под фактическое распределение участников Mobius.
